35. How Do Profiles Merge When There Are Conflicting IDs?

How Profiles Merge When There Are Conflicting IDs

Overview

When customer data comes from multiple sources with conflicting information, Adobe Experience Platform uses merge policies to determine which data takes priority and how profile fragments are combined into a single unified customer view.

Understanding Profile Fragments vs. Merged Profiles

Profile Fragments

  • Profile information for just one identity out of all the identities that exist for a customer
  • Each fragment represents data from a single dataset for a single identity
  • Multiple fragments can exist for the same customer across different datasets

Merged Profiles

  • The unified view created when multiple profile fragments are combined
  • Based on a defined merge policy that resolves conflicts
  • The total number of merged profiles is always lower than the total number of fragments

Key Point: Each merged profile is typically composed of multiple fragments from multiple datasets.

The Conflict Scenario: A Real Example

Let's say a customer interacts with your brand across two channels, and you have conflicting data:

Loyalty System Data:

```

First Name: Daniel

City: New York City

```

CRM System Data:

```

First Name: Danny

City: Portland

```

Questions:

  • Which first name wins? Daniel or Danny?
  • Which city wins? New York City or Portland?
  • How does Platform decide?

How Conflicts Are Resolved: Merge Policies

A merge policy is a set of rules that tells Experience Platform which data to prioritize when fragments conflict.

Two Merge Methods Available

1️⃣ Timestamp Ordered (Default)

How It Works:

  • Gives priority to the profile fragment that was updated most recently
  • The most recent data automatically "wins" in a conflict
  • No manual ordering required

Example:

```json

{

"attributeMerge": {

"type": "timestampOrdered"

}

}

```

Scenario:

  • CRM data (Danny, Portland) was uploaded after Loyalty data (Daniel, New York City)
  • Result: Danny and Portland win because the CRM data has a more recent timestamp

When to Use:

  • When you trust the most recently updated source to have the correct data
  • When data naturally gets cleaner/more accurate over time

Visual Confirmation in UI:

2️⃣ Dataset Precedence

How It Works:

  • You manually define the priority order of datasets
  • Conflicts are resolved based on which dataset ranks higher in your custom order
  • The dataset with highest priority always wins, regardless of timestamps

Example:

```json

{

"attributeMerge": {

"type": "dataSetPrecedence",

"order": [

"dataSetId_2", // Highest priority

"dataSetId_3", // Second priority

"dataSetId_1", // Third priority

"dataSetId_4" // Lowest priority

]

}

}

```

Scenario:

  • You decide Loyalty data is your source of truth (highest priority)
  • Even if CRM was updated more recently, Loyalty data wins
  • Result: Daniel and New York City win because Loyalty dataset has priority

When to Use:

  • When one system is more trusted or authoritative than others
  • When you have a business rule like "Loyalty data is always correct"
  • When you want to explicitly control which source wins

Important: Datasets not listed in the `order` array will not be merged — they must be explicitly included.

Real-World Decision Example

Scenario: Customer Moves

A customer moves from Portland to New York City:

| Data Point | Loyalty System | CRM System | Timestamp Ordered Result | Dataset Precedence Result (if Loyalty is Priority) |

|---|---|---|---|---|

| First Name | Daniel | Danny | Danny (CRM newer) | Daniel (Loyalty priority) |

| City | Portland | New York City | New York City (CRM newer) | Portland (Loyalty priority) |

Interpretation with Timestamp Ordered:

  • CRM is considered the most recent source, so its data wins
  • This makes sense if CRM was just updated with the customer's new address

Interpretation with Dataset Precedence (Loyalty priority):

  • Loyalty is your source of truth, so its data always wins
  • Useful if you've verified Loyalty data is cleaner/more accurate

How Identity Graph Affects Merging

When Identity Graph Creates Multiple Identities

If a customer has multiple identities (email, phone, loyalty ID, etc.), the identity graph determines which identities belong to the same person. The merge policy then decides how to handle conflicting attributes across those identities.

The attributeMerge Object

When conflicts arise between profile attributes during merging, the `attributeMerge` object specifies resolution:

```json

{

"attributeMerge": {

"type": "{ATTRIBUTE_MERGE_TYPE}"

}

}

```

  • `timestampOrdered` = Priority goes to most recently updated profile
  • `dataSetPrecedence` = Priority follows your manually defined dataset order

Union Schemas

When multiple schemas are enabled for profile, Experience Platform automatically creates a Union Schema for all profile-enabled record schemas sharing the same base class.

Key Point: Union Schemas don't exist for ExperienceEvent data because:

  • ExperienceEvent data is time-series based
  • Each event has a timestamp and unique ID
  • Collisions are not a problem — events don't conflict the way profile attributes do

Important Considerations for Merging

Batch Segmentation & Merges Don't Immediately Update

When two profiles merge together:

  • Changes will appear immediately in profile lookups
  • Changes will NOT appear in segment membership (`segmentMembership` attribute) until the next batch segmentation job runs

Example:

  • You have two mutually exclusive audiences: "Lives in Washington" and "Does NOT live in Washington"
  • Profile 1: Lives in Washington → qualifies for Audience A
  • Profile 2: Lives in Oregon → qualifies for Audience B
  • After profiles merge (but before next batch job): The merged profile is in both audiences until batch segmentation re-evaluates (typically next day)

Interview Answer Structure

Quick Version (1-2 minutes)

> "When profiles have conflicting IDs and data, Experience Platform uses merge policies to resolve conflicts. There are two main methods:

> 1. Timestamp Ordered — The most recently updated fragment wins. This is the default and works well when data naturally improves over time.

> 2. Dataset Precedence — You manually rank datasets by importance, and the highest-priority dataset always wins regardless of timing. This is useful when one source (like Loyalty) is your source of truth.

> For example, if Loyalty says 'Daniel in New York' and CRM says 'Danny in Portland,' timestamp ordered would pick CRM data if it was updated more recently. But if you set Loyalty as priority using dataset precedence, Loyalty data always wins."
