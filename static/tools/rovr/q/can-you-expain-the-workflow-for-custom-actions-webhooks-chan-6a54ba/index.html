Custom Actions & Webhooks Workflow in Adobe Journey Optimizer

Custom actions and webhooks allow you to connect Journey Optimizer to any third-party system via REST API calls, enabling you to trigger external services, send data to external platforms, or extend journeys beyond built-in messaging channels.

What Are Custom Actions?

Custom actions are connections configured by technical users that allow Journey Optimizer to call any third-party service or internal endpoint through a REST API with a JSON-formatted payload. Once configured, they appear in the Action category of the journey canvas palette and can be used by marketers to build journeys.

Common use cases include:

  • Sending messages via third-party providers (Epsilon, Firebase, Facebook, etc.)
  • Triggering external business processes
  • Writing data to external CRMs or data warehouses
  • Calling Adobe Developer endpoints

> Important: The configuration of custom actions must be performed by a technical user.

How It Works — High Level

```

AJO Journey → Custom Action Node → REST API Call (JSON payload)

→ Third-Party External System → Response (optional)

→ Response data available in journey for next steps

```

Phase 1: Configure the Custom Action

This is done by a technical administrator or developer.

Prerequisites

  • The target external endpoint must be accessible via REST API.
  • You must know the authentication method required by the endpoint.
  • The endpoint should ideally be a private or internal endpoint with appropriate capping or throttling limits in place.

Steps to Create a Custom Action

1. Navigate to Administration → Configurations → Actions.

2. Click Create action.

3. Enter a name and optional description for the action.

4. Select Action type → Custom.

5. Configure the URL section:

  • Set the HTTP method: `POST`, `PUT`, `PATCH`, `DELETE`, or `GET`
  • Enter the endpoint URL (static or with dynamic path variables using `{}` notation)

6. Configure Authentication:

  • No authentication — for open endpoints
  • Basic authentication — username/password
  • Custom authentication — using a data source to fetch tokens dynamically (e.g., OAuth 2.0 token refresh)
  • API Key — pass a key via header or query parameter

7. Configure Headers:

  • Add any required static or dynamic HTTP headers (e.g., `Content-Type: application/json`, authorization tokens)

8. Configure the Payload:

  • Define the request payload structure in JSON format
  • Mark each field as either:
  • Constant — always sends the same fixed value
  • Variable — value will be mapped by the marketer in the journey canvas

9. Optionally configure the Response payload:

  • Define the expected JSON response structure from the external system
  • Response values become available as variables in subsequent journey steps

> Note: Scalar arrays are supported in response payloads (e.g., `["val1", "val2"]`), but heterogeneous arrays are not supported (e.g., mixing numbers, strings, and booleans in the same array).

10. Set Capping/Throttling limits to protect the external endpoint from being overwhelmed.

11. Click Save.

Phase 2: Use the Custom Action in a Journey

Once configured, the custom action is available to marketers in the journey canvas.

Steps

1. Open or create a Journey in Journey Optimizer.

2. In the left palette, find your custom action under the Action category.

3. Drag and drop the custom action node onto the journey canvas at the appropriate step.

4. Click on the custom action node to open its configuration panel.

5. Map the variable fields defined in the payload:

  • Use the expression editor to map profile attributes, journey context data, event fields, or computed values to each variable parameter
  • Constant fields are pre-filled and cannot be changed here

6. If a response payload was defined, the response fields are now available as variables to use in:

  • Conditions (branch logic)
  • Personalization in subsequent message steps
  • Other downstream action parameters

Phase 3: Webhooks for Feedback & Notifications (Inbound)

In addition to outbound custom actions, AJO supports inbound webhooks for receiving real-time feedback from external systems (e.g., delivery status from a messaging provider).

Create a Feedback Webhook

1. Navigate to Administration → Channels → Feedback webhook settings.

2. Click Create Feedback Webhook.

3. In the Basic Configuration section, provide:

  • Webhook Name — a descriptive name
  • Channels — select Email and/or SMS
  • Webhook URL — the HTTPS endpoint where feedback events will be delivered

4. In the Authentication section, select:

  • No Authentication — no authentication headers added
  • JWT Authentication — provide JWT details if your endpoint requires it

5. In the Header Parameters section, configure any additional custom headers to send with each webhook request.

6. Click Submit.

> Note: You can edit a webhook at any time by opening it from the inventory and clicking Edit.

Phase 4: Monitor & Troubleshoot Custom Actions

Monitoring

  • Journey reports include custom action performance data showing execution counts, errors, and latency.
  • Navigate to Reports → your journey report to view custom action metrics.

Troubleshooting

  • Check the journey step events in AEP Query Service or the journey live view to inspect individual profile executions through the custom action node.
  • Common issues include:
  • Authentication failures — verify credentials and token refresh logic
  • Payload mapping errors — ensure all required variable fields are mapped
  • Timeout errors — the external endpoint may be slow or unreachable
  • Capping limit reached — the external endpoint's call limit has been hit

Key Considerations

| Topic | Detail |

|---|---|

| Who configures | Technical admin/developer configures; marketer uses in journeys |

| Protocol | REST API only, JSON payload format |

| Authentication | None, Basic, API Key, or Custom (OAuth) |

| Response payload | Supported — response values usable in subsequent journey steps |

| Scalar arrays | Supported in response payloads |

| Heterogeneous arrays | Not supported in response payloads |

| Capping/Throttling | Should be configured to protect external endpoints |

| Endpoint type | Private/internal endpoints recommended |

| Adobe Campaign | Native integration available for AC v7/v8 (separate from custom actions) |

Summary Flow

```

Technical Config → Define URL + Auth + Headers + Payload + Response

→ Marketer uses action node in Journey

→ Map profile/event data to variable fields

→ Journey executes → REST API call fires

→ External system receives data

→ Response parsed → available for downstream steps

→ Monitor via Journey Reports

```
