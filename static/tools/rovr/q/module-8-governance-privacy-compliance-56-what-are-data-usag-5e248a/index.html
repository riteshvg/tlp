Module 8: Governance, Privacy & Compliance 56. What are Data Usage Labels? Why are they important?

Data Usage Labels in Adobe Experience Platform

What are Data Usage Labels?

Data Usage Labels are metadata tags applied to datasets and schema fields in Adobe Experience Platform that categorize data according to the type of governance policies, privacy regulations, and compliance restrictions that apply to that data.

They are the foundation of the AEP Data Governance framework — without labels, the system has no way of knowing what type of data it is handling or what restrictions should apply.

> In simple terms: Data Usage Labels are like warning tags on data that tell the system — and the people using it — exactly what that data is, how sensitive it is, and what it can or cannot be used for.

The Three Label Categories

AEP provides three main categories of out-of-the-box labels:

1. 📄 Contract (C) Labels — Legal & Business Restrictions

Contract labels categorize data that has contractual obligations or is related to customer data governance policies. They define what you are legally or contractually allowed to do with the data.

| Label | Name | Meaning |

|---|---|---|

| C1 | No third-party export | Data cannot be shared with third parties |

| C2 | No export outside AEP | Data cannot leave the platform |

| C3 | No combining with on-site data | Cannot merge with website behavioral data |

| C4 | No on-site ad targeting | Cannot use for ads shown on your website |

| C5 | No cross-site targeting | Cannot use for retargeting across websites |

| C6 | No on-site personalization | Cannot personalize website experiences |

| C7 | No content personalization | Cannot personalize email/SMS/other content |

| C8 | No measurement | Cannot be used for analytics on websites or apps |

| C9 | No ML/AI usage | Cannot be used in machine learning models |

| C10 | No geofencing | Cannot be used for location-based targeting |

| C11 | No interest-based marketing | Cannot use for behavioral advertising |

| C12 | No general data export | Cannot be exported or downloaded from CJA/AEP |

2. 🪪 Identity (I) Labels — Personally Identifiable Information

Identity labels categorize data that can identify or contact a specific person. These are your PII flags.

| Label | Name | Meaning |

|---|---|---|

| I1 | Directly identifiable | Directly identifies a person (name, email, SSN, phone) |

| I2 | Indirectly identifiable | Can identify a person when combined with other data (device ID + location) |

3. ⚠️ Sensitive (S) Labels — Sensitive Data Categories

Sensitive labels categorize data that is considered highly sensitive — often regulated by privacy laws such as HIPAA, PCI-DSS, and GDPR.

| Label | Name | Meaning |

|---|---|---|

| S1 | Sensitive financial data | Bank accounts, credit cards, financial records |

| S2 | Sensitive health data | Medical records, health conditions, prescriptions |

| S3 | Sensitive location data | Precise geographic/location data |

Special Labels of Note in Customer Journey Analytics

Two labels have specific significance in Customer Journey Analytics:

| Label | Name | Impact in CJA |

|---|---|---|

| C8 | No measurement | Data cannot be used for analytics on websites or apps — blocks CJA reporting |

| C12 | No general data export | Data cannot be exported or downloaded from CJA via reporting, CSV, API, etc. |

> ⚠️ Data usage labels are not automatically propagated to stitched datasets in CJA — they must be added manually.

Where Labels Are Applied

Labels can be applied at two levels in AEP:

Level 1: Dataset-Level Labels

When you apply a label to an entire dataset, it is automatically propagated to ALL fields within that dataset.

```

Dataset: Customer_Master [Labels: C2, I1, S1]

├── Field: email → Inherits [C2, I1, S1]

├── Field: phone → Inherits [C2, I1, S1]

├── Field: name → Inherits [C2, I1, S1]

└── Field: purchase_history → Inherits [C2, I1, S1]

```

Level 2: Field-Level Labels

Labels can also be applied to individual fields without affecting other fields in the same dataset:

```

Dataset: Customer_Master [No dataset-level label]

├── Field: email → [I1, C2] ← PII label only on email

├── Field: phone → [I1] ← Identity label only on phone

├── Field: preferences → [C11] ← Interest-based restriction

└── Field: transaction_id → [No labels] ← Not sensitive

```

This gives you granular, field-level governance control.

How Labels Are Applied in the UI

Applying Labels to a Dataset:

Prerequisites:

  • You must have Manage Data Labels permission in Adobe Admin Console
  • Labels can be applied at any time — during ingestion or after

Steps:

1. In the AEP left navigation, select Datasets

2. Open the dataset you want to label (e.g., `Luma Loyalty Dataset`)

3. Click the Data Governance tab

4. Click the pencil (edit) icon next to the dataset name or field

5. In the modal, expand the label category:

  • Contract Labels → Check relevant C labels (e.g., C2)
  • Identity Labels → Check I1 or I2
  • Sensitive Labels → Check S1, S2, or S3

6. Click Save changes

7. Toggle Show inherited labels to confirm propagation across all fields

Applying Labels via the Policy Service API:

```http

PUT https://platform.adobe.io/data/foundation/dataset/datasets/{DATASET_ID}/labels

```

Request Body:

```json

{

"labels": ["C1", "C2", "I1"],

"optionalLabels": [

{

"option": {

"id": "https://ns.adobe.com/{TENANT_ID}/schemas/field-path",

"contentType": "application/vnd.adobe.xed-full+json;version=1",

"schemaPath": "/properties/personalEmail"

},

"labels": ["I1", "S1"]

}

]

}

```

Looking Up Labels via API:

```http

GET https://platform.adobe.io/data/foundation/dataset/datasets/{DATASET_ID}/labels

```

Response:

```json

{

"AEP:dataset:5abd49645591445e1ba04f87": {

"imsOrg": "{ORG_ID}",

"labels": ["C1", "C2", "C3", "I1", "I2"],

"optionalLabels": [

{

"option": {

"schemaPath": "/properties/repositoryCreatedBy"

},

"labels": ["S1", "S2"]

}

]

}

}

```

Viewing Available Labels

To see all available labels in your organization:

1. In the AEP UI, select Policies in the left navigation

2. Click the Labels tab

3. You will see:

  • All out-of-the-box Adobe-provided labels
  • Any custom labels your organization has created
  • The Friendly Name and short abbreviation for each label

4. To create a custom label, click Create label

Custom Labels

In addition to Adobe's out-of-the-box labels, you can define your own custom labels for your organization's specific needs.

When to Use Custom Labels:

  • Your organization has internal data classification standards not covered by Adobe defaults
  • You need to enforce industry-specific restrictions (e.g., financial services, healthcare)
  • You want to create labels aligned with internal data ownership or regional regulations

Creating a Custom Label:

1. Navigate to Policies → Labels

2. Click Create label

3. Provide:

  • Name — short identifier (e.g., `EMEA_ONLY`)
  • Friendly Name — descriptive name (e.g., "EMEA Region Data Only")
  • Description — what this label means

4. Click Save

Label Inheritance for Audience Segments

When you build an audience using data from labeled datasets, the audience automatically inherits those labels:

```

Audience: "High Value Customers in Europe"

Built from:

├── Dataset A → Labels: [C2, I1]

├── Dataset B → Labels: [C11, S1]

└── Dataset C → Labels: [No labels]

Resulting Audience Inherited Labels: [C2, I1, C11, S1]

When activated to destination:

├── System checks if [C2, I1, C11, S1] violates any policies

├── C11 restriction + Behavioral Advertising → ❌ BLOCKED

└── Resolution: Change destination type or remove C11-labeled data

```

Why Data Usage Labels Are Important

1. 🛡️ Regulatory Compliance

Labels help you comply with global privacy regulations:

| Regulation | Applicable Labels | Protection |

|---|---|---|

| GDPR | I1, I2 | PII protection for EU citizens |

| HIPAA | S2 | Health data protection |

| PCI-DSS | S1 | Payment card data protection |

| CCPA/CPRA | I1, I2, C11 | California consumer rights |

| LGPD | I1, I2 | Brazilian data privacy law |

2. ⚙️ Automatic Enforcement

Labels trigger automatic policy enforcement at activation time — no manual checking required:

```

Without Labels:

❌ No controls → Wrong data activated → Data breach → Regulatory fines

With Labels:

✅ Label detected → Policy checked → Violation? → Blocked automatically

```

3. 📊 Operational Clarity

Labels create organization-wide clarity about what data is and how it can be used:

  • New team members instantly know data restrictions
  • Data stewards can audit usage across the organization
  • Legal and compliance teams have confidence in data controls

4. 🔍 Auditability

Labels create an audit trail demonstrating:

  • Which data was used for which purposes
  • Which activations were blocked and why
  • Compliance evidence for regulators

5. 🚀 Flexibility Without Risk

Labels allow teams to move fast while staying compliant:

  • Marketers can self-serve audience activation
  • System automatically prevents policy violations
  • No need for legal review on every activation

The Complete Governance Framework

Data usage labels don't work alone — they are part of a complete framework:

```

Step 1: LABEL

Apply data usage labels to datasets and fields

(C labels, I labels, S labels, custom labels)

↓

Step 2: DEFINE POLICIES

Create marketing action policies that reference labels

("If data has C11 label, block behavioral advertising")

↓

Step 3: ENFORCE

AEP automatically evaluates labels + policies at activation time

↓

Step 4: AUDIT

Review policy violations, blocked activations, and data usage history

↓

Step 5: ITERATE

Update labels and policies as regulations and business needs change

```

Summary

| Aspect | Detail |

|---|---|

| What they are | Metadata tags categorizing data by type and restriction |

| Three categories | Contract (C), Identity (I), Sensitive (S) |

| Where applied | Dataset level (propagates to all fields) or Field level (granular) |

| Custom labels | Yes — organizations can create their own |

| Inheritance | Audiences inherit labels from underlying datasets |

| Enforcement | Automatic at activation time via policies |

| Key CJA labels | C8 (no measurement), C12 (no export) |

| Availability | All AEP organizations — no special license required |

Data Usage Labels are the cornerstone of AEP's data governance strategy — they are the first line of defense in ensuring that your organization handles customer data responsibly, compliantly, and with full accountability across every team and use case.
