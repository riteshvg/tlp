53. What are Consent Policies? How do they enforce restrictions?

Consent Policies in Adobe Experience Platform

What are Consent Policies?

Consent Policies are rules that filter which profiles can be activated to destinations based on the customer's consent or preference choices. They ensure that your marketing communications and data usage respect what individual customers have opted into (or out of).

Consent policies are part of AEP's broader data governance framework and work alongside data usage labels and marketing action policies to enforce compliance.

Key Concept:

> A consent policy says: "Only send this audience to this destination IF the profile has given consent for this specific purpose."

Consent Policies vs. Data Governance Policies

These are two distinct types of policies in AEP:

| Type | Focus | Example |

|---|---|---|

| Data Governance Policy | Restricts activation based on data labels and marketing actions | "Cannot use PII (I1 label) for behavioral advertising (C11 action)" |

| Consent Policy | Restricts activation based on individual customer consent choices | "Only activate to email if customer consented to email marketing" |

Both must pass for activation to succeed:

```

Attempt to Activate Audience to Destination

↓

Check Data Governance Policies → ✅ Pass or ❌ Fail

↓

Check Consent Policies → ✅ Pass or ❌ Fail

↓

Both Pass? → ✅ Activation Proceeds

Either Fails? → ❌ Activation BLOCKED

```

Important: Consent Policy Availability

⚠️ Consent policies and automatic enforcement are ONLY available for organizations that have purchased:

  • Adobe Healthcare Shield, OR
  • Adobe Privacy & Security Shield

If your organization doesn't have these add-ons, you can still use data governance policies, but not consent policies.

How Consent Policies Work

The Two-Part Structure

Every consent policy has two logical components:

Part 1: The "IF" Condition

This defines when the policy is evaluated:

```

IF [condition is met]

```

The condition can be based on:

  • A specific marketing action being performed (e.g., "email marketing")
  • Specific data usage labels present on the data (e.g., "I1 - Directly Identifiable")
  • A combination of marketing actions AND labels using AND/OR logic

Example Conditions:

  • "IF the marketing action is 'Email Marketing'"
  • "IF the data has label 'I1' AND the action is 'Behavioral Advertising'"
  • "IF the action is 'Direct Mail' OR 'SMS Marketing'"
Part 2: The "THEN" Consequence

This defines what consent attributes must be present for the profile to be included:

```

THEN [require these consent attributes]

```

The policy checks for the presence of XDM consent data fields in the profile's consent preferences.

Example Consequences:

  • "THEN require `consents.marketing.email.val` = true"
  • "THEN require `consents.marketing.push.val` = true"
  • "THEN require `personalization.content.val` = true"

Real-World Example

Scenario: Email Marketing Policy

```

Policy Name: "Email Marketing Consent Required"

IF: Marketing action = "Email Marketing"

THEN: Profile must have consents.marketing.email.val = true

How it Works:

├── Profile A: email consent = true → ✅ Include in email audience

├── Profile B: email consent = false → ❌ Exclude from email audience

└── Profile C: email consent = unknown → ❌ Exclude (conservative approach)

```

Scenario: Multi-Channel Consent

```

Policy Name: "Multi-Channel Marketing Compliance"

IF: Marketing action = "Direct Marketing"

THEN: Profile must have:

AND consents.marketing.email.val = true

OR consents.marketing.push.val = true

How it Works:

├── Profile A: email=true, push=false → ✅ Include (has email consent)

├── Profile B: email=false, push=true → ✅ Include (has push consent)

├── Profile C: email=false, push=false → ❌ Exclude (has neither)

└── Profile D: email=unknown, push=false → ❌ Exclude (conservative)

```

Creating a Consent Policy

Step 1: Define the "IF" Condition

In the Consent Policy UI:

1. Click the "If" section

2. Select the marketing actions and/or data usage labels that should trigger the policy

3. Use the dropdown between multiple conditions to select AND or OR logic

Available Marketing Actions include:

  • Email Marketing
  • SMS Marketing
  • Push Notifications
  • Direct Mail
  • Behavioral Advertising
  • On-Site Personalization
  • Mobile App Analytics
  • And more...

Available Data Usage Labels include:

  • C1, C2, C3... (Contract labels)
  • I1, I2 (Identity labels)
  • S1, S2 (Sensitive labels)

Step 2: Select Consent Attributes

In the "Then" section:

1. Choose which consent attributes must be present

2. These are XDM consent data fields like:

  • `consents.marketing.email.val`
  • `consents.marketing.push.val`
  • `consents.marketing.sms.val`
  • `personalization.content.val`
  • `personalization.marketing.preferred_channel.val`

3. You can use AND/OR logic to combine multiple consent attributes

Step 3: Name & Save

1. Give the policy a descriptive name: "Email Marketing Consent Required"

2. Add an optional description

3. Click Save

The policy is now active and will be enforced on all destination activations.

Consent Policy Enforcement During Activation

What Happens When You Activate an Audience

```

1. User selects audience for destination activation

↓

2. System retrieves all active Consent Policies

↓

3. For each profile in the audience:

a. Check if any policy's "IF" condition applies

b. If yes, check if profile has required "THEN" consent attributes

c. Profile fails check? → EXCLUDE from activation

↓

4. Final audience = [Original audience] - [Profiles failing consent checks]

↓

5. Only compliant profiles are sent to destination

```

Example Enforcement

Policy: "Email Marketing Consent Required"

  • IF: Marketing action = Email Marketing
  • THEN: consents.marketing.email.val = true

Audience: "Newsletter Subscribers" (1,000 profiles)

Activation to: Email Marketing Platform

What Gets Activated:

  • ✅ 750 profiles WITH email consent = true → SENT
  • ❌ 250 profiles WITH email consent = false → EXCLUDED
  • ❌ Unknown/missing consent → EXCLUDED (conservative)

Result: Only 750 profiles delivered to email platform

Default Consent Behavior

By default, if a profile has opted out of communications, it is automatically excluded from subsequent deliveries — even without an explicit consent policy.

However, you can create a custom consent policy to override this default if needed.

Example: Override Default

Scenario: A customer opted out of email, but you have legal basis to send them transactional emails (e.g., order confirmations).

Solution: Create a consent policy:

  • IF: Marketing action = "Transactional Email"
  • THEN: No consent check required (allow all profiles)

This creates a carve-out for transactional messages while respecting opt-out preferences for marketing.
