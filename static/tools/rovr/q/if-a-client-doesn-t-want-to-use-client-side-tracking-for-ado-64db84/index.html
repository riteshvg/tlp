Alternatives to Client-Side Tracking for Adobe Analytics

If a client wants to avoid client-side tracking (i.e., no JavaScript tags or SDKs deployed in the browser), there are several server-side alternatives:

Option 1: Edge Network Server API

As discussed earlier, the Edge Network Server API is the most modern approach for server-side data collection:

  • Data is sent directly from your backend servers to the Adobe Edge Network
  • No client-side code required in the browser
  • Uses XDM-formatted payloads routed via a configured Datastream to Analytics
  • Ideal for:
  • IoT devices
  • Desktop applications
  • Server-to-server integrations
  • Sensitive data environments

Option 2: Adobe Analytics Data Insertion API

The Data Insertion API is a legacy server-side method specifically for Adobe Analytics:

  • Allows you to send data directly to Adobe Analytics collection servers via HTTP requests
  • No JavaScript/AppMeasurement required on the page
  • Uses XML or query string parameters formatted to Analytics collection standards
  • Ideal for:
  • Environments where JavaScript cannot run
  • Server-generated pages
  • Non-browser environments

Option 3: Bulk Data Insertion API (BDIA)

The Bulk Data Insertion API is the modern, recommended replacement for the legacy Data Insertion API:

  • Allows you to upload large batches of Analytics hits via CSV files
  • Processes data in bulk rather than real-time hit-by-hit
  • Supports client hints data (as confirmed in the retrieved context)
  • Ideal for:
  • Migrating historical data
  • High-volume server-side environments
  • Offline data processing scenarios

Option 4: Data Sources

Data Sources is another method for importing offline or server-side data into Adobe Analytics:

  • Upload data via CSV files through the Analytics UI or FTP
  • Useful for importing offline transaction data, call center data, or CRM data
  • Not a real-time method — data is processed in batches

Comparison of Server-Side Methods

| Method | Real-Time | Format | Best For |

|---|---|---|---|

| Edge Network Server API | ✓ Yes | XDM/JSON | Modern server-side, IoT, backend systems |

| Data Insertion API | ✓ Yes | XML/Query String | Legacy server-side, non-browser environments |

| Bulk Data Insertion API | ✗ Batch | CSV | High-volume batch processing |

| Data Sources | ✗ Batch | CSV | Offline data imports |

Important Considerations

Identity & Visitor Tracking

When moving away from client-side tracking, one of the biggest challenges is visitor identity:

  • Client-side tracking uses cookies (e.g., AMCV cookie, s_vi cookie) to persist visitor IDs
  • Server-side implementations must manually manage and pass visitor/device identifiers in each API call
  • You will need to determine how to generate, store, and pass a consistent Experience Cloud ID (ECID) or custom visitor ID

Data Completeness

  • Client-side tracking automatically captures many dimensions (browser type, screen resolution, referrer, etc.)
  • Server-side implementations require you to explicitly pass all desired data points in your API calls
  • Some dimensions may be unavailable or require additional effort to capture

Privacy & Compliance

  • Server-side tracking can be more privacy-friendly as it doesn't rely on browser cookies
  • However, you still need to ensure compliance with GDPR, CCPA, and other regulations
  • Consent management must still be implemented at the application level

Recommended Modern Approach

For clients wanting a fully server-side, modern implementation, the recommended path is:

```

Your Backend System / Server

↓

Edge Network Server API (XDM-formatted data)

↓

Adobe Experience Platform Edge Network

↓

Datastream Routes to:

→ Adobe Analytics

→ Adobe Experience Platform

→ Audience Manager

→ Event Forwarding Destinations

```

This gives the most flexibility, scalability, and future-proofing while avoiding any client-side code.
