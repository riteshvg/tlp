Overview

The Adobe Experience Platform Edge Network is the globally distributed, real-time data processing layer that sits between your Web SDK implementation and all Adobe services. It is the central nervous system of the modern Adobe data collection architecture.

> Key Definition: The Edge Network is a network of globally distributed servers that receives data from Web SDK, processes it in real-time, routes it to Adobe services, and returns personalization decisions — all within milliseconds.

Core Roles of the Edge Network

The Edge Network plays seven critical roles in a Web SDK implementation:

Role 1: Global, Low-Latency Data Reception

The Edge Network is geographically distributed across the globe to minimize latency:

```

Global Edge Network Nodes:

──────────────────────────

Americas:

• Oregon, USA (us-or2)

• Virginia, USA (us-va5)

• São Paulo, Brazil

Europe:

• London, UK (irl1)

• Amsterdam, Netherlands

• Frankfurt, Germany

Asia Pacific:

• Singapore (sgp3)

• Tokyo, Japan

• Sydney, Australia

• Mumbai, India

How It Works:

─────────────

1. Customer in London fires Web SDK event

2. DNS routes request to nearest Edge node (London)

3. London Edge node processes request in ~5-10ms

4. Response returned within ~100ms total

5. Customer in Tokyo routed to Tokyo node instead

```

Latency Comparison:

| Architecture | Typical Latency | Why |

|---|---|---|

| Legacy (AppMeasurement) | 200-500ms per beacon | Multiple separate library calls |

| Web SDK + Edge Network | 50-150ms total | Single consolidated request |

| Edge Segmentation | <10ms | On-node processing |

Role 2: Single Endpoint Consolidation

Before Edge Network, websites needed multiple tracking calls to different endpoints:

Legacy Architecture (Multiple Calls):

```

Browser fires:

Call 1: → omniture.com (Adobe Analytics beacon)

Call 2: → tt.omtrdc.net (Adobe Target mbox request)

Call 3: → cm.everesttech.net (Audience Manager)

Call 4: → demdex.net (Visitor ID service)

Total: 4 separate network calls

4 separate round trips

4 separate processing pipelines

Performance impact on page load

```

Modern Architecture (Single Edge Network Call):

```

Browser fires:

Single Call: → adobedc.net (Edge Network)

Handles ALL services internally

Edge Network routes to:

→ AEP (profile update, event ingestion)

→ Analytics (beacon processing)

→ Target (personalization decision)

→ AJO (journey triggering)

Total: 1 network call

1 round trip

Multiple services served

Minimal performance impact

```

Web SDK single request payload:

```javascript

// ONE sendEvent call handles everything

alloy("sendEvent", {

xdm: {

eventType: "web.webpagedetails.pageViews",

web: {

webPageDetails: {

name: "Homepage",

URL: "https://www.retailstore.com"

}

},

identityMap: {

ECID: [{"id": "98765432109876543210"}]

}

},

renderDecisions: true // Gets Target AND AJO decisions

})

.then(function(result) {

// Contains:

// - Analytics confirmation

// - Target personalization decisions

// - AJO offer decisions

// - Identity resolution results

// All in ONE response

});

```

Role 3: Real-Time Identity Resolution

The Edge Network performs identity resolution on every incoming event:

```

Incoming Event Identity Map:

─────────────────────────────

{

"identityMap": {

"ECID": [{"id": "98765432109876543210"}],

"Email": [{"id": "jane@email.com", "authenticatedState": "authenticated"}]

}

}

Edge Network Identity Processing:

──────────────────────────────────

Step 1: Look up ECID: 98765432109876543210

Result: Found in edge profile cluster or2

Step 2: Look up Email: jane@email.com

Result: Found in identity graph, linked to CRM_ID: CRM-123

Step 3: Link ECID + Email + CRM_ID

Result: Single unified identity graph updated

Step 4: Retrieve edge profile for personalization

Result: Jane's profile attributes loaded for Target/AJO

Step 5: Return identity confirmation in response

Result: {

"type": "identity:result",

"payload": [{

"id": "98765432109876543210",

"namespace": {"code": "ECID"}

}]

}

```

Identity Cookie Management:

The Edge Network manages identity cookies on behalf of Web SDK:

```

Cookies Set by Edge Network:

────────────────────────────

ECID Cookie: kndctr_ORGID_AdobeOrg_identity

Value: Encrypted ECID + metadata

Expiry: 2 years

Domain: First-party domain (if configured)

Cluster Cookie: kndctr_ORGID_AdobeOrg_cluster

Value: or2 (Edge cluster identifier)

Expiry: 30 minutes

Purpose: Routes subsequent requests to same Edge node

(ensures session consistency)

Consent Cookie: kndctr_ORGID_AdobeOrg_consent

Value: Encoded consent preferences

Purpose: Persists consent across sessions

```

Role 5: Real-Time Personalization Decisioning

The Edge Network enables sub-100ms personalization by processing Target and AJO decisions at the edge:

```

Personalization Flow:

─────────────────────

1. Web SDK sends event with renderDecisions: true

↓

2. Edge Network receives event

↓

3. Edge profile retrieved (from edge cache)

↓

4. Edge segmentation evaluated (milliseconds)

↓

5. Target/AJO queried for applicable activities/offers

↓

6. Winning experiences determined based on:

• Audience membership

• Activity priority

• Traffic allocation

• Profile attributes

↓

7. Personalization decisions packaged in response

↓

8. Web SDK receives response (~100ms after request)

↓

9. Web SDK renders personalized content automatically

(if renderDecisions: true)

```

Role 6: Edge Segmentation

The Edge Network performs real-time audience evaluation at the edge level:

```

Edge Segmentation Processing:

──────────────────────────────

Event: commerce.productListAdds (customer adds item to cart)

↓

Edge Network evaluates edge-enabled segments:

Segment 1: "Active Cart - No Purchase"

Criteria: productListAdds event in last 30 minutes

AND no commerce.purchases event

Evaluation: ✅ Jane QUALIFIES

Result: Segment membership updated in edge profile

Segment 2: "Gold Loyalty Members"

Criteria: loyalty.tier == "Gold"

Evaluation: ✅ Jane QUALIFIES (from edge profile)

Result: Already qualified, no change

Segment 3: "High Value Cart ($200+)"

Criteria: commerce.order.priceTotal >= 200

Evaluation: ❌ Jane's cart = $129.99

Result: Jane does NOT qualify

Segment Results Applied:

→ Available for Target personalization immediately

→ Available for AJO journey entry conditions

→ No batch processing wait time

→ Real-time decision making enabled

```

Segment Types by Processing Location:

| Segment Type | Where Evaluated | Latency | Use Case |

|---|---|---|---|

| Edge Segmentation | Edge Network node | Milliseconds | Real-time personalization |

| Streaming Segmentation | AEP hub | Seconds | Journey triggering |

| Batch Segmentation | AEP hub | Hours | Large-scale audience building |

Role 7: Data Transformation & Enrichment

The Edge Network applies Data Prep transformations and enriches data before routing to services:

```

Data Transformation Flow:

──────────────────────────

Input XDM Data:

{

"eventType": "commerce.productViews",

"productListItems": [{

"SKU": "shoe-run-001",

"priceTotal": 129.99

}]

}

Data Prep Transformations Applied:

1. UPPERCASE("shoe-run-001") → "SHOE-RUN-001"

2. ROUND(129.99, 0) → 130

3. Map eventType → Analytics event1

4. Map SKU → Analytics eVar10

5. Add geolocation: City = "San Francisco" (from IP)

6. Add device type: "Mobile" (from user agent)

Output to Analytics:

{

"event1": 1,

"eVar10": "SHOE-RUN-001",

"currency": 130,

"pageName": "Product Detail Page",

"prop10": "Mobile",

"prop20": "San Francisco"

}

Output to AEP:

{

"eventType": "commerce.productViews",

"productListItems": [{

"SKU": "SHOE-RUN-001", // Uppercase applied

"priceTotal": 130, // Rounded

"normalizedSKU": "SHOE-RUN-001"

}],

"placeContext": {

"geo": {

"city": "San Francisco", // Enriched from IP

"countryCode": "US"

}

},

"device": {

"type": "Mobile" // Enriched from user agent

}

}

```
