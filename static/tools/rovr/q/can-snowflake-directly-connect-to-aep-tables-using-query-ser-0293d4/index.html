Snowflake Connecting to AEP via Query Service DB Credentials

This is a nuanced topic — let me break down exactly what's possible, what's not, and what the architecture actually looks like based on the available documentation.

What the Documentation Covers

The retrieved context describes two different integration patterns between Snowflake and AEP — and it's important to understand the direction of data flow:

Option 1: AEP → Snowflake (Destination / Data Share)

(AEP pushes data TO Snowflake)

This is a Snowflake destination in AEP that uses Snowflake Data Sharing (not Query Service). Key characteristics:

  • No data is physically exported — Adobe grants read-only access to a live dynamic table hosted in Adobe's own Snowflake environment
  • You query the shared table from your Snowflake account directly
  • Data has a fixed 7-day TTL (Time-to-Live) — the dynamic table expires 7 days after last export
  • If you need to retain data longer than 7 days, you must copy it into a table you own in your Snowflake instance before expiry
  • Deleting the dataflow in AEP causes the dynamic table to disappear from your Snowflake account
  • You must accept a private listing from Adobe the first time the dataflow is set up

> This pattern is likely closest to your use case of "Snowflake pulling daily data from AEP datasets."

Option 2: Snowflake → AEP (Source Connector)

(Snowflake pushes data TO AEP)

This is the Snowflake source connector — the reverse direction from what you need. Not relevant to your question.

Regarding Query Service DB Credentials + Snowflake

Based on the documentation, Query Service DB credentials are designed for third-party SQL clients (e.g., Power BI, Tableau, DbVisualizer, Postico) connecting into AEP's Query Service endpoint — not for Snowflake to natively pull from AEP datasets as if they were Snowflake-native tables.

Key points:

  • Query Service supports SSL modes for third-party client connections (e.g., `verify-full`)
  • It supports attribute-based access control to restrict column/dataset-level access
  • However, the documentation does not describe Snowflake itself as a supported Query Service client in the same way as BI tools
  • The `INSERT OVERWRITE INTO` command is not supported in Query Service, which would limit certain write-back patterns

> The documentation explicitly notes: "The Snowflake source is available to users who have purchased Real-Time Customer Data Platform Ultimate."

Recommended Architecture for Your Use Case

If your goal is Snowflake pulling daily AEP dataset data, the most supported path based on the documentation is:

1. Set up a Snowflake Destination dataflow in AEP (AEP UI → Destinations → Snowflake)

2. AEP shares audience/dataset data as a dynamic table in Adobe's Snowflake environment

3. Your Snowflake account queries or copies that shared table into your own Snowflake tables (critical — do this before the 7-day TTL expires)

4. Accept the private listing from Adobe in your Snowflake account on first setup

What I'd Recommend Clarifying with Adobe

The documentation does not explicitly confirm or deny whether a Snowflake instance can authenticate against Query Service using a DB credential the same way a BI tool would. I'd recommend:

  • Confirming with your Adobe account team whether Snowflake's external table or data sharing features can leverage Query Service credentials
  • Discussing whether Data Distiller's export capabilities (CTAS + cloud export) might be a better fit if you need more control over scheduling and retention than the 7-day TTL allows
