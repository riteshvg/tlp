Overview

Adobe Experience Platform processes data in two distinct paths:

1. Edge Network – Geographically distributed servers for real-time, low-latency processing

2. AEP Hub – Centralized data center for comprehensive, thorough data processing

Understanding when to use each path is fundamental to AEP architecture design.

> Core Principle:

> - Edge Network = Optimized for SPEED (behavioral data, real-time personalization)

> - AEP Hub = Optimized for THOROUGHNESS (complete data, rich profiles, batch processing)

Side-by-Side Comparison

| Aspect | Edge Network | AEP Hub |

|---|---|---|

| Primary Optimization | Speed | Thoroughness |

| Location | Globally distributed (close to user) | Centrally located data center |

| Latency | Milliseconds to ~100ms | Seconds to minutes |

| Data Processed | Behavioral data only | All data types |

| Data Stored | Audience memberships, profile identities, personalization attributes | All historical data, complete profiles |

| Profile Type | Edge Profile (lightweight) | Full Real-Time Customer Profile |

| Segmentation | Edge Segmentation | Batch + Streaming Segmentation |

| Use Case | Real-time personalization, tag management | Analytics, ML, batch activation |

| API Access | Edge Network API | AEP APIs (Profile, Segmentation, etc.) |

Edge Network: Deep Dive

What Lives on the Edge Network

The Edge Network only stores what is needed for real-time decisions:

```

Data Stored on Edge Network:

─────────────────────────────

✅ Audience Memberships

→ Which segments a profile currently qualifies for

→ Used for real-time personalization targeting

✅ Profile Identities

→ ECID, email, CRM ID (key identifiers only)

→ Used for identity resolution at edge

✅ Personalization Attributes

→ Key profile attributes needed for decisions

→ Example: loyaltyTier, preferredLanguage, country

→ Subset of full hub profile

✅ Consent Data

→ Customer consent preferences

→ Determines if data can be collected

❌ NOT Stored on Edge:

→ Full event history

→ Complete attribute set

→ All identity relationships

→ Raw unprocessed data

→ Historical behavioral data

```

Edge Network Processing Pipeline

```

1. REQUEST RECEIVED

├── Validate datastreamId

├── Authenticate organization

└── Check rate limits

2. IDENTITY RESOLUTION

├── Read ECID from cookie

├── Extract identity map from payload

├── Look up identity cluster

└── Route to correct edge node (for session continuity)

3. EDGE PROFILE RETRIEVAL

├── Load edge profile for resolved identity

├── Get audience memberships

└── Get personalization attributes

4. DATA TRANSFORMATION

├── Apply Data Prep field mappings

├── Enrich with geolocation (from IP)

├── Enrich with device data (from user agent)

└── Apply calculated fields

5. EDGE SEGMENTATION

├── Evaluate edge-enabled segments

├── Update audience memberships

└── Make updated memberships available for step 6

6. PERSONALIZATION DECISIONING

├── Query Target for activity assignments

├── Query AJO for offer decisions

├── Apply audience memberships to decisions

└── Select winning experiences

7. SERVICE ROUTING (ASYNC)

├── Route to AEP streaming ingestion

├── Route to Analytics beacon processing

├── Route to AJO journey triggering

└── Route to event forwarding destinations

8. RESPONSE ASSEMBLY

├── Package personalization decisions

├── Include identity results

├── Include consent confirmations

└── Return to Web SDK (~100ms)

```

Services Available on Edge Network

| Service | Description | Use Case |

|---|---|---|

| Edge Segmentation | Real-time segment evaluation at edge | Instant audience qualification |

| Edge Profiles | Lightweight profile cache at edge | Sub-millisecond profile lookups |

| Edge Destinations | Real-time audience activation | Synchronous personalization |

| Data Collection | Receive and process Web/Mobile SDK data | All data collection |

| SDK Extensions | Process extension-specific logic | Tags extensions |

| Event Forwarding | Server-side tag forwarding | Third-party integrations |

| Tag Management | Deploy and manage tags | Website tag deployment |

AEP Hub: Deep Dive

What Lives on AEP Hub

The Hub is the central repository for all data — no limits on completeness:

```

Data Stored on AEP Hub:

────────────────────────

✅ Complete Data Lake

→ All raw events ever ingested

→ Full event history for every customer

→ All batch data from source connectors

→ Queryable via Query Service

✅ Full Real-Time Customer Profile

→ All profile attributes from all datasets

→ Complete experience event timeline

→ Computed attributes (pre-calculated metrics)

→ Full segment membership history

✅ Complete Identity Graph

→ All identity relationships across all namespaces

→ Full identity history and link events

→ Privacy deletion records

✅ ML Model Training Data

→ Customer AI propensity scores

→ Attribution AI models

→ Custom ML model inputs/outputs

✅ Governance & Compliance

→ Data usage labels

→ Policy enforcement records

→ Consent histories

→ Audit logs

```

Hub Processing Pipeline

```

1. STREAMING INGESTION

├── Receive from Edge Network (async)

├── Receive from Streaming API (direct)

└── Validate XDM schema compliance

2. BATCH INGESTION

├── Source connector pull (Salesforce, Marketo, etc.)

├── File upload (CSV, JSON, Parquet)

└── Cloud storage sync (S3, Azure Blob, SFTP)

3. DATA LAKE STORAGE

├── Store raw event data

├── Organize into datasets

├── Apply data governance labels

└── Index for Query Service access

4. IDENTITY GRAPH UPDATES

├── Process new identity links

├── Resolve identity graph conflicts

├── Apply identity optimization algorithm

└── Update identity graph (full resolution)

5. PROFILE UPDATES

├── Merge new data with existing profile

├── Apply merge policies

├── Update computed attributes

└── Publish updated profile snapshot

6. STREAMING SEGMENTATION

├── Evaluate streaming-eligible segments

├── Update segment membership in real-time

└── Trigger AJO journey entries

7. BATCH SEGMENTATION (DAILY)

├── Evaluate all profiles against all segments

├── Generate complete audience membership snapshots

└── Export to destinations

8. ACTIVATION

├── Export audiences to destinations

├── Trigger AJO campaigns

└── Update RTCDP audiences

```

When to Use Edge Network vs. Hub

Use Edge Network When:

```

Scenario 1: Real-Time Web Personalization

Customer visits homepage

→ Need personalized hero banner in <100ms

→ Use: Edge Network + Target/AJO

→ Why: Edge profile + edge segmentation = instant decisions

Scenario 2: A/B Testing

Customer loads product page

→ Need A/B test assignment before page renders

→ Use: Edge Network + Target

→ Why: Edge decision = no flicker, instant assignment

Scenario 3: Live Event Tracking

Customer clicks "Add to Cart" button

→ Need to track behavioral event immediately

→ Use: Web SDK → Edge Network → AEP

→ Why: Edge Network processes and routes in real-time

Scenario 4: Server-Side Tag Forwarding

Customer triggers event

→ Need to forward to Google Analytics, Snowflake, CRM

→ Use: Edge Network Event Forwarding

→ Why: Server-side is faster and more reliable than client-side

```

Use AEP Hub When:

```

Scenario 1: Batch Audience Building

Marketing team needs "all customers who spent $500+ in last 90 days"

→ Use: Hub Batch Segmentation + Query Service

→ Why: Requires full event history analysis

Scenario 2: ML Propensity Scoring

Need to predict which customers will churn in next 30 days

→ Use: Customer AI (Hub)

→ Why: Requires months of behavioral history for training

Scenario 3: Attribution Analysis

Need to understand which marketing channels drive revenue

→ Use: Attribution AI + Query Service (Hub)

→ Why: Requires complete multi-touch event history

Scenario 4: CRM Data Integration

Need to ingest 5 years of customer purchase history from Salesforce

→ Use: Salesforce Source Connector → Hub Batch Ingestion

→ Why: Batch data ingestion, not real-time

Scenario 5: Ad Audience Activation

Need to push "Loyal Customers" segment to Google Ads

→ Use: Hub Destinations

→ Why: Batch audience export doesn't require real-time

```

Data Flow Between Edge Network and Hub

The Edge Network and Hub work together — not as alternatives:

```

Complete Data Flow:

────────────────────

STEP 1: Edge Network receives event

Web SDK → Edge Network Node (nearest)

Latency: ~5-10ms

STEP 2: Edge Network processes for real-time needs

• Identity resolution at edge

• Edge profile update

• Edge segmentation evaluation

• Personalization decisioning

Latency: ~50-100ms

STEP 3: Edge Network returns real-time response

• Personalization decisions

• Identity confirmation

• Consent confirmation

Returned to Web SDK: ~100ms total

STEP 4: Edge Network routes data to Hub (ASYNC)

• Data forwarded to AEP streaming ingestion

• Does NOT wait for hub processing

• Customer already has personalization response

Latency from customer's perspective: 0ms additional

STEP 5: Hub processes event comprehensively

• Full identity graph update

• Complete profile update

• Streaming segmentation

• Data lake storage

Latency from event receipt: 1-30 seconds

STEP 6: Hub updates edge profile (if needed)

• Updated audience memberships pushed to edge

• New personalization attributes available at edge

• Next customer visit uses updated edge profile

Propagation: Minutes

```
