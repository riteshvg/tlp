Can I modify data elements or rules dynamically using the _satellite object, or is it read-only?

Great question! The short answer is: technically you can modify it, but it is not supported, not recommended, and changes are temporary and session-scoped only.

What's Actually Possible (Technically)

Since `_satellite._container` is just a JavaScript object in memory, the browser does not enforce true immutability on it. You can manipulate it:

Overriding a Data Element Value at Runtime

The `_satellite` object provides a semi-supported way to override data element values using:

```javascript

_satellite.setVar('myDataElement', 'overrideValue');

```

Then when any rule calls `_satellite.getVar('myDataElement')`, it will return your overridden value for that session only.

> This is actually a documented, legitimate use of the `_satellite` API — it is the intended way to set custom/JavaScript variable-type data elements dynamically.

Directly Mutating `_satellite._container` (Unsupported)

You could technically do things like:

```javascript

// Forcibly overwrite a data element definition

_satellite._container.dataElements['myDE'].settings.path = 'newPath';

// Or push a fake rule into the rules array

_satellite._container.rules.push({ name: 'fakeRule', ... });

```

But this is entirely unsupported and comes with serious problems (see below).

Why You Should NOT Directly Mutate `_satellite._container`

| Risk | Detail |

|---|---|

| Temporary only | All changes exist only in the current browser session's memory. A page refresh wipes everything back to the published library state. |

| No persistence | Changes never propagate back to the Tags property in Adobe Data Collection UI. |

| Unpredictable behavior | Rules are compiled and evaluated in a specific internal structure. Injecting malformed or unexpected objects can cause silent failures or JavaScript errors. |

| Library version dependency | Since `_satellite._container` is a private/internal API, its structure can change with any Tags runtime library update, breaking any code that depends on it. |

| Security concern | If your site allows arbitrary JS execution that mutates `_satellite`, malicious actors could hijack your data collection logic (e.g., redirect beacon calls). |

The Right Ways to Dynamically Influence Tag Behavior

If you need dynamic behavior, Adobe Tags provides proper mechanisms:

1. Use `_satellite.setVar()` for Dynamic Values

```javascript

// Set before a rule fires

_satellite.setVar('pageName', dynamicValue);

```

2. Use Custom Code Data Elements

Define a Custom Code data element in the Tags UI that reads from a JS variable your app controls:

```javascript

// Data element custom code

return window.myApp.dynamicValue;

```

3. Use Direct Call Rules

Trigger specific rules programmatically from your application code:

```javascript

_satellite.track('myDirectCallEvent', { payload: data });

```

4. Use the Data Layer

Push values into a Adobe Client Data Layer (ACDL) or standard `digitalData` object, and have data elements point to those locations — giving you full dynamic control cleanly.

Summary

| Action | Supported? | Persistent? |

|---|---|---|

| `_satellite.setVar()` | ✅ Yes, documented | ❌ Session only |

| `_satellite.track()` | ✅ Yes, documented | N/A |

| `_satellite.getVar()` | ✅ Yes, documented | N/A |

| Mutating `_satellite._container` directly | ⛔ No, unsupported | ❌ Session only |

| Modifying rules/data elements permanently | ⛔ Only via Tags UI/API | ✅ Requires publish |

Bottom line: Use `_satellite.setVar()` and `_satellite.track()` for legitimate runtime dynamism. Leave `_satellite._container` alone outside of read-only debugging scenarios.
