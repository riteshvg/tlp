Overview

The complete data flow from Web SDK to Activation represents the end-to-end journey of customer data through the Adobe Experience Platform ecosystem. Understanding this flow is critical for architects, developers, and solution consultants implementing AEP-based solutions.

```

┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐

│ WEB SDK │───▶│ EDGE NETWORK │───▶│ AEP │───▶│ PROFILE │───▶│ ACTIVATION │

│ (Browser) │ │ (Global) │ │ (Platform) │ │ (Store) │ │ (Channels) │

└──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘ └──────────────┘

```

Stage 1: Web SDK (Browser/Client Side)

What Happens at This Stage

The Web SDK (`alloy.js`) is the client-side JavaScript library that captures customer interactions and sends them to the Edge Network.

How Web SDK Works

Installation & Initialization

The Web SDK can be deployed via:

  • Adobe Tags (Launch) – Web SDK tag extension (recommended)
  • NPM package – `npm install @adobe/alloy`
  • CDN script tag – Direct script include

Initialization example:

```javascript

// Configure Web SDK

alloy("configure", {

datastreamId: "YOUR_DATASTREAM_ID",

orgId: "YOUR_ORG_ID@AdobeOrg",

onBeforeEventSend: function(options) {

// Modify event data before sending

return options;

}

});

```

Data Capture

Web SDK captures data through:

1. Page View Events:

```javascript

alloy("sendEvent", {

xdm: {

eventType: "web.webpagedetails.pageViews",

web: {

webPageDetails: {

name: "Home Page",

URL: "https://www.example.com"

}

}

}

});

```

2. Interaction Events:

```javascript

alloy("sendEvent", {

xdm: {

eventType: "commerce.productViews",

commerce: {

productViews: {

value: 1

}

},

productListItems: [{

SKU: "SKU-123",

name: "Running Shoes",

priceTotal: 89.99

}]

}

});

```

3. Identity Events (Login):

```javascript

// Get current ECID first

alloy("getIdentity").then(function(result) {

alloy("sendEvent", {

xdm: {

eventType: "userAccount.login",

identityMap: {

Email: [{

id: "customer@example.com",

authenticatedState: "authenticated",

primary: true

}],

ECID: [{

id: result.identity.ECID,

authenticatedState: "ambiguous",

primary: false

}]

}

}

});

});

```

What Web SDK Sends to Edge Network

Every Web SDK event includes:

| Component | Description |

|---|---|

| XDM payload | Structured event data in XDM format |

| Identity map | Customer identities (ECID, email, etc.) |

| Datastream ID | Routing configuration identifier |

| Request context | Device, browser, location metadata |

| Consent data | Customer consent preferences |

Web SDK Key Features

  • Single consolidated library – Replaces AppMeasurement, AT.js, and Visitor API
  • XDM-native – All data structured in Experience Data Model format
  • Low latency – Sends data directly to nearest Edge Network node
  • Personalization support – Receives real-time decisions from Target/AJO
  • Automatic ECID management – Handles identity cookie creation and management

Stage 2: Edge Network (Global Processing Layer)

What Happens at This Stage

The Edge Network is a globally distributed network of servers that receives Web SDK data, processes it in real-time, and routes it to appropriate Adobe services.

Edge Network Architecture

```

Customer Browser

↓ (HTTPS POST to adobedc.net)

┌─────────────────────────────────────────┐

│ EDGE NETWORK NODE │

│ (Geographically closest to customer) │

│ │

│ 1. Receive & validate request │

│ 2. Apply Datastream configuration │

│ 3. Run Data Prep transformations │

│ 4. Identity resolution │

│ 5. Edge segmentation │

│ 6. Personalization decisioning │

│ 7. Route data to hub services │

│ 8. Return personalization response │

└─────────────────────────────────────────┘

↓ (Route to AEP Hub)

```

Key Edge Network Processing Steps

Step 1: Request Reception & Validation
  • Receives HTTPS POST request from Web SDK at `adobedc.net`
  • Validates datastream ID and organization credentials
  • Checks request format and XDM schema compliance
  • Applies bot detection filtering
Step 2: Datastream Configuration Applied

The datastream configuration determines:

  • Which Adobe services receive the data
  • Which event forwarding rules apply
  • What data transformations to perform
  • Which sandboxes to route data to

Datastream routing example:

| Service | Enabled | Configuration |

|---|---|---|

| Adobe Experience Platform | ✅ | Dataset, sandbox, profile opt-in |

| Adobe Analytics | ✅ | Report suite mapping |

| Adobe Target | ✅ | Property token, decisioning method |

| Adobe Audience Manager | ✅ | Cookie destinations, URL destinations |

Step 3: Data Preparation (Field Mapping)
  • Data Prep transforms and maps fields from XDM to solution-specific formats
  • Maps XDM fields to Analytics props/eVars
  • Applies calculated fields and conditional transformations
  • Handles data type conversions

```

XDM Field → Target Field

─────────────────────────────────────────────────

commerce.productViews.value → Analytics event1

productListItems[0].SKU → Analytics eVar5

web.webPageDetails.name → Analytics pageName

```

Step 4: Identity Resolution at Edge
  • ECID generation – Creates/reads ECID cookie for anonymous visitors
  • Identity linking – Links authenticated identities with anonymous ECID
  • Profile lookup – Retrieves edge profile for real-time decisioning
Step 5: Edge Segmentation
  • Evaluates edge-enabled segments against the current event
  • Updates audience membership in milliseconds
  • Makes segment membership available for real-time personalization
Step 6: Personalization Decisioning
  • Queries Adobe Target for A/B test assignments
  • Retrieves AJO decisions for offer management
  • Returns personalization payload to Web SDK for rendering
Step 7: Data Routing to Hub

Routes processed data to:

  • AEP Data Lake – For storage and analytics
  • Identity Service – For identity graph updates
  • Real-Time Profile – For profile updates
  • Adobe Analytics – For reporting
  • Event Forwarding – For server-side tag forwarding

Edge Network Response

The Edge Network returns a single consolidated response to Web SDK containing:

```json

{

"requestId": "abc123",

"handle": [

{

"type": "identity:result",

"payload": [{

"id": "12345678901234567890",

"namespace": { "code": "ECID" }

}]

},

{

"type": "personalization:decisions",

"payload": [{

"id": "offer-abc123",

"scope": "homepage-hero",

"items": [{

"id": "item-xyz",

"data": {

"content": "<h1>Welcome back!</h1>"

}

}]

}]

},

{

"type": "activation:push",

"payload": []

}

]

}

```

Stage 3: Adobe Experience Platform (Data Processing)

What Happens at This Stage

Data routed from Edge Network to AEP Hub undergoes comprehensive processing and storage.

3a. Streaming Ingestion

Data arrives in AEP via Streaming Ingestion:

  • Latency: Sub-second to seconds
  • Protocol: HTTP POST to AEP Streaming Ingestion API
  • Format: XDM-formatted JSON
  • Validation: Schema compliance checking

```

Edge Network

↓ (Streaming)

AEP Streaming Ingestion API

↓

XDM Validation & Processing

↓

Data Lake Storage

```

3b. Data Lake Storage

All ingested data is stored in the AEP Data Lake:

  • Raw event storage – Complete XDM event records
  • Dataset organization – Events organized by dataset
  • Queryable – Accessible via Query Service
  • Retention policies – Configurable data retention periods

3c. Identity Service Processing

Identity Service processes every ingested event:

1. Extracts identity map from the event payload

2. Looks up existing identities in the identity graph

3. Links new identities to existing graph nodes

4. Updates identity graph with new connections

5. Publishes identity events for downstream processing

```

Event arrives with:

Email: customer@example.com

ECID: 12345678901234567890

Identity Service:

1. Finds existing ECID node in graph

2. Finds existing Email node in graph

3. Links Email ↔ ECID in identity graph

4. Single unified identity graph created

```

3d. Real-Time Profile Processing

Real-Time Customer Profile service:

1. Receives identity-resolved event

2. Looks up unified profile using resolved identity

3. Merges new data according to merge policy

4. Updates profile attributes with latest values

5. Appends experience event to profile timeline

6. Triggers streaming segmentation evaluation

```

Profile Update Flow:

New Event → Identity Resolution → Profile Lookup

↓

Merge Policy Applied → Profile Updated

↓

Streaming Segmentation → Audience Membership Updated

↓

Computed Attributes → Recalculated if triggered

```

3e. Segmentation Processing

Segmentation Service evaluates the updated profile:

| Type | When | Latency |

|---|---|---|

| Streaming | As each event arrives | Seconds |

| Edge | At Edge Network | Milliseconds |

| Batch | Scheduled (daily) | Hours |

Streaming segmentation example:

```

Profile receives "commerce.purchases" event

↓

Segmentation evaluates: "Customers who made a purchase in last 7 days"

↓

Profile qualifies for segment → Audience membership updated

↓

AJO journey trigger fires → Customer enters post-purchase journey

```

Stage 4: Real-Time Customer Profile

What Happens at This Stage

The Real-Time Customer Profile store maintains a unified, always-current view of each customer.

Profile Structure

```json

{

"identityMap": {

"Email": [{"id": "customer@example.com"}],

"ECID": [{"id": "12345678901234567890"}],

"CRM_ID": [{"id": "CRM-12345"}]

},

"attributes": {

"person": {

"name": {

"firstName": "Jane",

"lastName": "Smith"

}

},

"personalEmail": {

"address": "customer@example.com"

},

"loyaltyTier": "Gold",

"lifetimeValue": 1250.00

},

"experienceEvents": [

{

"eventType": "commerce.purchases",

"timestamp": "2024-01-15T10:30:00Z",

"commerce": {

"order": {

"priceTotal": 89.99

}

}

}

],

"segmentMembership": {

"High Value Customers": {

"status": "realized",

"lastQualificationTime": "2024-01-15T10:30:05Z"

},

"Recent Purchasers": {

"status": "realized",

"lastQualificationTime": "2024-01-15T10:30:05Z"

}

}

}

```

Profile Merge Policies

Merge policies define how conflicting data is resolved:

| Policy Type | Description |

|---|---|

| Dataset precedence | Specific datasets take priority |

| Timestamp ordered | Most recent data wins |

| Custom | Business-defined rules |

Computed Attributes

Pre-calculated values derived from event history:

```

totalPurchaseValue = SUM(commerce.order.priceTotal) last 90 days

purchaseCount = COUNT(commerce.purchases) last 30 days

daysSinceLastPurchase = DAYS_SINCE(last commerce.purchases event)

```

Stage 5: Activation

What Happens at This Stage

Activated audiences and profile data are delivered to downstream channels for personalization, messaging, and advertising.

5a. Adobe Journey Optimizer (AJO)

Event-triggered journeys:

```

Profile qualifies for "Recent Purchasers" segment

↓

AJO Journey triggered by segment qualification event

↓

Journey evaluates conditions (loyalty tier, purchase value, etc.)

↓

Email action: Send personalized thank you email

↓

Wait 3 days

↓

Check: Did customer view recommended products?

↓

YES → Send product recommendation email

NO → Send discount offer SMS

```

5b. Adobe Target (Real-Time Personalization)

Synchronous personalization during the same Web SDK call:

```

Web SDK sendEvent()

↓

Edge Network processes request

↓

Target retrieves audience membership from edge profile

↓

Selects winning experience for A/B test

↓

Returns personalization decision in Edge Network response

↓

Web SDK renders personalized content (< 100ms)

```

5c. Real-Time CDP Destinations

Audience activation to external platforms:

```

Profile qualifies for "High Value Customers" segment

↓

RTCDP streaming destination activated

↓

Profile data exported to:

  • Google Ads (suppression/targeting)
  • Facebook Custom Audiences
  • LinkedIn Matched Audiences
  • Salesforce C
