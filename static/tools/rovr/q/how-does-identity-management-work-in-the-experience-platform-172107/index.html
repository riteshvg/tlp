How does identity management work in the Experience Platform Web SDK?

Identity Management in the Experience Platform Web SDK

Overview

Identity management in the Experience Platform Web SDK centers on accurately identifying users across devices, sessions, and channels. The Web SDK handles identity through a combination of automatic identity generation, custom identity configuration, and identity stitching — all feeding into the Adobe Experience Platform Identity Service and Real-Time Customer Profile.

The Foundation: Identity Namespaces

Identity namespaces provide context for what an identifier represents. Without a namespace, a value like `12345` is meaningless — it could be a CRM ID, a loyalty number, or anything else. Namespaces disambiguate identity values across systems.

Standard Namespaces

AEP provides built-in namespaces including:

  • ECID — Experience Cloud ID (cookie-based, automatically managed by Web SDK)
  • Email — Hashed or plain email addresses
  • Phone — Phone numbers
  • AAID — Adobe Analytics ID
  • IDFA / GAID — Mobile device identifiers

Custom Namespaces

You can create your own namespaces for proprietary identifiers:

Prerequisites: Access to the Adobe Experience Platform UI or Data Collection UI

1. In the Data Collection interface, select [!UICONTROL Identities] from the left-rail navigation

2. Select [!UICONTROL Create identity namespace]

3. Provide a [!UICONTROL Display name] (e.g., `Luma CRM ID`) and an [!UICONTROL Identity symbol] value (e.g., `lumaCRMId`)

4. Select the identity type — [!UICONTROL Cross-device ID] for person-level identifiers like CRM IDs

5. Select [!UICONTROL Create]

> ⚠️ Critical Configuration Warning: Choosing the wrong identity type causes serious graph corruption:

> - Person-level identifiers (CRM IDs, email) → must use Cross-device ID

> - Device/browser identifiers (cookies, device IDs) → must use Cookie ID or Device ID

>

> Misconfiguration requires manual correction by Adobe Engineering and cannot be self-remedied in production.

The ECID: Web SDK's Automatic Identity

The Experience Cloud ID (ECID) is the cornerstone of Web SDK identity management.

How the ECID Works

  • The Web SDK automatically generates and manages the ECID — no manual configuration required
  • It is stored in a first-party cookie in the user's browser
  • It persists across sessions for the same browser
  • Every Web SDK event automatically includes the ECID
  • It serves as the anonymous identifier that links browsing behavior before a user authenticates

ECID Lifecycle

```

First Visit → ECID Generated → Stored in Cookie

↓

Subsequent Visits → ECID Retrieved from Cookie → Sent with Every Event

↓

User Authenticates → CRM ID Linked to ECID → Identity Graph Updated

↓

Cross-Device Recognition → Unified Profile Built

```

Mobile SDK ECID Behavior

On mobile, the ECID works differently:

  • The Mobile SDK generates a unique ECID when the app is first installed
  • It is stored in persistent memory on the device
  • It is sent with every hit
  • It is removed when the user uninstalls the app or sets global privacy status to opt out
  • Reinstalling the app creates a new profile with a new unique ECID

The identityMap: Sending Custom Identities

The `identityMap` is the primary mechanism for passing custom identity data through the Web SDK.

What is identityMap?

`identityMap` is a special XDM field automatically included in all schemas using the XDM ExperienceEvent class. It is:

  • A must-have object for any web-related data collection
  • The container for the ECID required for web user identification
  • The key mechanism for setting internal customer IDs for authenticated users

identityMap Structure

```javascript

alloy("sendEvent", {

xdm: {

identityMap: {

// Namespace symbol as key

"lumaCRMId": [

{

id: "ABC123", // The identity value

authenticatedState: "authenticated", // Authentication state

primary: true // Primary identity flag

}

],

"Email": [

{

id: "user@example.com",

authenticatedState: "authenticated",

primary: false

}

]

}

}

});

```

Authentication States

Every identity in the `identityMap` must declare an authenticated state:

| State | Description | When to Use |

|---|---|---|

| `authenticated` | User has actively logged in during this session | Post-login events |

| `ambiguous` | Identity present but authentication not confirmed | Default/unknown state |

| `loggedOut` | User has explicitly logged out | Post-logout events |

> ✅ Best Practice: Only set `authenticatedState: "authenticated"` after a user has actively logged in during the current session. This ensures identity stitching only occurs for verified users.

Primary Identity Flag

  • Only one identity per event should be marked `primary: true`
  • The primary identity is used as the main lookup key for the Real-Time Customer Profile
  • If no primary identity is explicitly set, the ECID is used as the primary identity by default

How Identity Stitching Works

Identity stitching (also called identity graph linking) connects anonymous and known identities into a unified profile.

The Stitching Process

1. Anonymous Phase — User browses the site; Web SDK sends events with only an ECID

2. Authentication Event — User logs in; Web SDK sends an event with both ECID and CRM ID in `identityMap`

3. Graph Linking — Identity Service detects both identifiers in the same event and creates a link between them

4. Profile Merging — All previous anonymous behavior (tied to the ECID) is now associated with the known customer profile (tied to the CRM ID)

5. Cross-Device Recognition — If the same customer logs in on another device, their identities are further linked

Practical Example

```

Day 1: Anonymous browsing

→ ECID: abc-123 (browser cookie)

→ Events recorded against ECID only

Day 3: User logs in

→ ECID: abc-123 + CRM ID: customer-456

→ Identity graph links abc-123 ↔ customer-456

→ All Day 1 behavior now attributed to customer-456

Day 5: Same user logs in on mobile

→ ECID: xyz-789 (mobile device) + CRM ID: customer-456

→ Identity graph: abc-123 ↔ customer-456 ↔ xyz-789

→ Unified cross-device profile created

```

Configuring Identities in Your Web SDK Implementation

Method 1: Via the Data Element (Tag-Based Implementation)

If you are using Adobe Tags (Launch) with the Web SDK extension:

1. In the Tags UI, create a new [!UICONTROL Data Element]

2. Select [!UICONTROL Adobe Experience Platform Web SDK] as the extension

3. Select [!UICONTROL Identity Map] as the data element type

4. Configure your namespace and identity value mappings

5. Reference this data element in your XDM object data element

Method 2: Directly in Web SDK JavaScript

Pass identity data directly in the `sendEvent` command:

```javascript

alloy("sendEvent", {

xdm: {

identityMap: {

"lumaCRMId": [{

id: getUserCRMId(), // Retrieve from your data layer

authenticatedState: "authenticated",

primary: true

}]

}

}

});

```

Method 3: Using the setConsent Command

For privacy-compliant identity management, use the `setConsent` command before sending identity data to respect user consent choices:

```javascript

alloy("setConsent", {

consent: [{

standard: "Adobe",

version: "2.0",

value: {

general: "in"

}

}]

});

```

Identity Management Integration Points

Identity data collected by the Web SDK flows into the broader AEP ecosystem:

| Downstream System | How Identity is Used |

|---|---|

| Identity Service | Builds and updates the identity graph |

| Real-Time Customer Profile | Merges data from all linked identities into one profile |

| Audience Segmentation | Evaluates profile membership based on unified identity |

| Adobe Target | Personalizes experiences using the unified profile |

| Adobe Analytics | Stitches visitor data across authenticated sessions |

| Adobe Journey Optimizer | Triggers journeys based on known customer identity |

| Destinations | Activates audiences using identity namespaces (e.g., hashed email) |

Key Best Practices

✅ Always include ECID — Let the Web SDK manage this automatically; never suppress it

✅ Use correct identity types — Person-level IDs must use Cross-device ID namespace type

✅ Set authenticated state accurately — Only mark identities as `authenticated` after confirmed login

✅ Designate one primary identity — Avoid multiple primary identities in a single event

✅ Respect consent — Only send known identity data after obtaining appropriate user consent

✅ Test with Assurance — Use Adobe Experience Platform Assurance to validate identity data is being sent correctly before going to production

✅ Plan your namespace strategy — Define all custom namespaces before implementation begins to avoid rework
