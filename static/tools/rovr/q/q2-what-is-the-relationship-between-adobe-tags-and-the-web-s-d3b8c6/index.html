Overview

The relationship between Adobe Tags and Web SDK (alloy.js) is best understood as:

> Adobe Tags is the DEPLOYMENT MECHANISM. Web SDK is the DATA COLLECTION ENGINE.

They are two separate but deeply integrated tools that work together to form the modern Adobe data collection stack.

```

┌─────────────────────────────────────────────────────────┐

│ ADOBE TAGS │

│ (Deployment & Configuration Layer) │

│ │

│ ┌─────────────────────────────────────────────────┐ │

│ │ WEB SDK EXTENSION (alloy.js) │ │

│ │ (Data Collection Engine) │ │

│ │ │ │

│ │ • Configure SDK settings │ │

│ │ • Define data collection rules │ │

│ │ • Map data elements to XDM │ │

│ │ • Send events to Edge Network │ │

│ └─────────────────────────────────────────────────┘ │

└─────────────────────────────────────────────────────────┘

↓

Edge Network → AEP → Profile → AJO

```

What is Adobe Tags in This Context?

Adobe Tags is the container and deployment system that:

  • Hosts the Web SDK JavaScript library on your website
  • Configures the Web SDK settings (datastream ID, org ID, etc.)
  • Controls when and how Web SDK events are fired
  • Maps website data layer values to XDM fields
  • Manages the deployment lifecycle (Dev → Staging → Production)

What is Web SDK (alloy.js) in This Context?

Web SDK is the core data collection library that:

  • Communicates with the Adobe Edge Network
  • Sends XDM-formatted events to Adobe services
  • Receives personalization decisions from Target/AJO
  • Manages visitor identity (ECID)
  • Handles consent management

Three Ways to Deploy Web SDK

Web SDK can be deployed in three ways — Tags being the most recommended:

| Deployment Method | Description | Recommended For |

|---|---|---|

| Via Adobe Tags ✅ | Web SDK deployed as a Tags extension | Most implementations — non-technical marketers |

| Via NPM Package | `npm install @adobe/alloy` in your codebase | Single-page apps, React/Angular/Vue |

| Via CDN Script Tag | Direct `<script>` include of alloy.js | Simple implementations, quick POCs |

Method 1: Web SDK Deployed Via Adobe Tags (Recommended)

How It Works

When you deploy Web SDK via Tags:

1. Tags loads the alloy.js library from Adobe CDN

2. Tags initializes Web SDK with your configuration

3. Tags rules fire Web SDK commands based on user interactions

4. Web SDK sends events to Edge Network

Step-by-Step Configuration

Step 1: Install the Web SDK Extension

1. Log in to experience.adobe.com

2. Navigate to Data Collection → Tags

3. Open your Tag Property

4. Go to Extensions → Catalog

5. Search for Adobe Experience Platform Web SDK

6. Click Install

Step 2: Configure the Web SDK Extension

After installation, configure the extension:

1. Navigate to Extensions → Installed

2. Click Configure on the Web SDK extension

3. Fill in the required settings:

Instance Configuration:

```

Instance Name: alloy (default)

IMS Org ID: YOUR_ORG_ID@AdobeOrg

Edge Domain: data.example.com (custom first-party domain)

```

Datastreams Configuration:

```

Production Environment: prod-datastream-id

Staging Environment: staging-datastream-id

Development Environment: dev-datastream-id

```

Privacy & Consent:

```

Default Consent: In / Out / Pending

```

Identity:

```

Migrate ECID from VisitorAPI: ✅ (if migrating from legacy)

Authenticate State: Ambiguous / Authenticated / Logged Out

```

Data Collection:

```

On Before Event Send Callback: (custom code to modify events)

On Before Link Click Send Callback: (for link tracking)

Click Collection: Enabled ✅

Download Links Qualifier: \.pdf$

```

Advanced Settings:

```

Edge Base Path: ee

```

Method 2: Web SDK Deployed Via NPM (Direct Implementation)

When Tags is not used, Web SDK is deployed directly in the website codebase:

```javascript

// Install: npm install @adobe/alloy

import { createInstance } from "@adobe/alloy";

const alloy = createInstance({ name: "alloy" });

// Configure Web SDK directly

alloy("configure", {

datastreamId: "YOUR_DATASTREAM_ID",

orgId: "YOUR_ORG_ID@AdobeOrg",

debugEnabled: false,

defaultConsent: "in",

onBeforeEventSend: function(options) {

// Modify XDM data before sending

options.xdm._retail = {

environment: "production"

};

return options;

}

});

// Send events directly in JavaScript

alloy("sendEvent", {

xdm: {

eventType: "web.webpagedetails.pageViews",

web: {

webPageDetails: {

name: document.title,

URL: window.location.href

}

}

}

});

```

Pros:

  • Full control over Web SDK initialization
  • Better integration with JavaScript frameworks (React, Angular, Vue)
  • No dependency on Tags

Cons:

  • Requires developer for every change
  • No self-service for marketing team
  • No visual rule builder

How Tags and Web SDK Work Together at Runtime

Here is exactly what happens when a visitor loads a page:

```

1. VISITOR ARRIVES ON WEBSITE

Browser requests www.retailstore.com

2. PAGE LOADS TAGS EMBED CODE

<script src="//assets.adobedtm.com/launch-prod-abc123.min.js">

3. TAGS LIBRARY LOADS

Tags JavaScript downloads and executes

All configured extensions initialize

4. WEB SDK EXTENSION INITIALIZES

alloy("configure", { datastreamId: "...", orgId: "..." })

ECID cookie read or created

Edge Network connection established

5. TAGS RULES EVALUATE

Rule: "Page View" → Event: Library Loaded → Conditions met

Action fires: Web SDK "Send Event"

6. WEB SDK SENDS XDM EVENT TO EDGE

POST https://data.example.com/ee/v2/interact

{

"events": [{

"xdm": {

"eventType": "web.webpagedetails.pageViews",

"web": { "webPageDetails": { "name": "Homepage" } },

"identityMap": { "ECID": [{"id": "98765432..."}] }

}

}]

}

7. EDGE NETWORK PROCESSES REQUEST

Routes to AEP, Analytics, Target per datastream config

8. EDGE NETWORK RETURNS RESPONSE

Personalization decisions, identity confirmation

9. TAGS PROCESSES RESPONSE

Web SDK extension renders personalization

Analytics beacon confirmed

Target experience applied

```

Comparison: Tags + Web SDK vs. Web SDK Alone

| Aspect | Tags + Web SDK | Web SDK Alone |

|---|---|---|

| Setup complexity | Moderate (UI-based) | Low (code-based) |

| Developer dependency | Low | High |

| Marketing self-service | ✅ Full | ❌ None |

| Rule-based logic | ✅ Visual builder | ❌ Code only |

| Version control | ✅ Built-in environments | ❌ Manual |

| Deployment speed | ✅ Minutes | ❌ Dev cycle |

| Approval workflow | ✅ Built-in | ❌ Manual |

| Debugging tools | ✅ Tags Debugger + Assurance | ⚠️ Assurance only |

| Framework compatibility | ⚠️ Limited SPA support | ✅ Full |

| Custom logic | ✅ Custom code actions | ✅ Full JavaScript |

Web SDK Actions Available in Tags Extension

The Web SDK Tags extension provides these action types:

| Action Type | Description | Use Case |

|---|---|---|

| Send Event | Sends XDM event to Edge Network | Page views, interactions, purchases |

| Apply Response | Handles Edge Network response | Personalization rendering |

| Set Consent | Updates customer consent | GDPR/CCPA compliance |

| Append Identity to URL | Adds identity to outbound links | Cross-domain tracking |

| Reset Identity | Clears stored identity | Logout scenarios |

| Get Identity | Retrieves current ECID | Identity-dependent operations |

| Send Media Event | Tracks media playback | Video/audio analytics |

| Create Media Session | Initializes media tracking | Streaming media |

| Update Variable | Modifies XDM variable | Building XDM incrementally |

| Send Image Beacon | Sends data via image request | Legacy compatibility |

| Redirect With Identity | Cross-domain redirect with identity | Multi-domain tracking |

Data Elements Available in Tags Extension

The Web SDK Tags extension provides these data element types:

| Data Element Type | Description |

|---|---|

| XDM Object | Build complete XDM payloads visually |

| Variable | Create reusable XDM variable objects |

| Identity Map | Define customer identity maps |

| Media Collection Instance Details | Media tracking configuration |
