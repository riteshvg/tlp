Overview

Configuring a datastream for production use requires careful planning, sequential service configuration, validation, and governance controls. A production datastream is the live configuration that processes real customer data and drives business-critical Adobe services.

Prerequisites Before Configuration

Before creating a production datastream, ensure the following are in place:

| Prerequisite | Description | Why Required |

|---|---|---|

| XDM Schema | Production XDM event schema created | Defines data structure for AEP ingestion |

| AEP Datasets | Event and profile datasets created | Data storage for ingested events |

| Identity Namespaces | Required namespaces configured | Enables identity resolution |

| Analytics Report Suite | Production report suite available | Required for Analytics service |

| Target Property Token | Production Target property created | Required for Target service |

| AJO Configuration | Journey Optimizer provisioned | Required for AJO service |

| Subdomain Delegation | First-party domain configured | Required for production identity |

| User Permissions | Datastream manage permissions | Required to create/edit datastreams |

Step-by-Step: Configure a Production Datastream

Step 1: Access Datastreams UI

1. Log in to experience.adobe.com

2. Navigate to Data Collection → Datastreams

3. Click New Datastream

Step 2: Basic Datastream Configuration

Fill in the foundational settings:

```

Name: "Retail Store - Production"

Description: "Production datastream for retail e-commerce website

and mobile app. Owner: Analytics Engineering Team."

Default Environment: Production

```

Best Practices for Naming:

```

Recommended Naming Convention:

[Brand] - [Environment] - [Property Type]

Examples:

✅ "RetailStore - Production - Web"

✅ "RetailStore - Production - Mobile iOS"

✅ "RetailStore - Staging - Web"

✅ "RetailStore - Development - Web"

Avoid:

❌ "Datastream 1"

❌ "Test"

❌ "My Datastream"

```

Step 4: Save Basic Configuration

Click Save to create the base datastream before adding services.

> Note: After saving changes to a datastream configuration, allow up to 35 minutes for changes to propagate across the Edge Network. During this propagation window, requests may still be served with the previous configuration.

Step 10: Configure Data Prep Mappings

Set up field transformations for production data quality:

1. Navigate to Data Prep for Data Collection in the AEP service

2. Click Add Mapping

3. Configure field mappings:

```

Production Data Prep Rules:

────────────────────────────

Rule 1: Email Normalization

Source: identityMap.Email[0].id

Transform: LOWERCASE(email)

Target: normalizedEmail

Rule 2: Product Price Formatting

Source: productListItems[0].priceTotal

Transform: ROUND(price, 2)

Target: formattedPrice

Rule 3: Timestamp Conversion

Source: timestamp

Transform: TO_UNIX_TIMESTAMP(timestamp)

Target: unixTimestamp

Rule 4: Country Code Normalization

Source: placeContext.geo.countryCode

Transform: UPPERCASE(countryCode)

Target: normalizedCountryCode

Rule 5: Currency Calculation

Source: commerce.order.priceTotal

Transform: MULTIPLY(priceTotal, 1.1) // Add 10% tax

Target: totalWithTax

```

Step 11: Configure Datastream Overrides

Datastream overrides allow you to change service configurations for specific events without creating separate datastreams:

When to Use Overrides:

```

Use Case 1: Different Dataset per Business Unit

Default Datastream: routes to "All Events Dataset"

Override: Marketing events → "Marketing Events Dataset"

Override: Commerce events → "Commerce Events Dataset"

Use Case 2: Different Report Suite per Region

Default: Global report suite

Override: US events → US report suite

Override: EU events → EU report suite

Use Case 3: Different Sandbox per Team

Default: Production sandbox

Override: Internal test traffic → Staging sandbox

```

Configuring Overrides:

```javascript

// Web SDK implementation with override

alloy("sendEvent", {

xdm: {

eventType: "commerce.purchases"

},

edgeConfigOverrides: {

com_adobe_experience_platform: {

datasets: {

event: {

datasetId: "commerce-events-dataset-id"

}

}

},

com_adobe_analytics: {

reportSuites: ["commerce-specific-rs"]

}

}

});

```

Step 12: Validate Production Configuration

Validation Checklist

| Validation Step | Method | Expected Result |

|---|---|---|

| Events reaching Edge Network | Assurance | Events visible in real-time |

| Analytics data flowing | Analytics Real-time Report | Hits appearing within seconds |

| Target personalization working | Target Activity QA | Correct experience rendered |

| AEP datasets populating | Dataset Activity Monitor | Batches ingesting successfully |

| AJO journeys triggering | Journey Live Reports | Profiles entering journeys |

| Identity graph forming | Identity Graph Viewer | Identities correctly linked |

| Profile updating | Profile Viewer | Attributes and events visible |

Using Assurance for Validation

1. Navigate to Data Collection → Assurance

2. Create a new Production session

3. Connect your production website

4. Verify events in Assurance:

```

Expected Events in Assurance:

✅ Web SDK Request/Response pairs

✅ Identity resolution events

✅ Analytics hit confirmation

✅ Target decisioning response

✅ AJO event forwarding confirmation

✅ AEP ingestion confirmation

```

Production Datastream vs. Development/Staging

| Setting | Development | Staging | Production |

|---|---|---|---|

| AEP Sandbox | Dev sandbox | Staging sandbox | Prod sandbox |

| AEP Dataset | Dev events dataset | Staging events dataset | Prod events dataset |

| Analytics Report Suite | Dev RS | Staging RS | Prod RS |

| Target Environment | Development | Staging | Production |

| Target Property | Dev property | Staging property | Prod property |

| Bot Detection | Optional | Enabled | ✅ Always enabled |

| Enhanced Encryption | Optional | Recommended | ✅ Required (if sensitive) |

| Audit Logging | Optional | Recommended | ✅ Always enabled |

| Access Control | Open | Restricted | ✅ Strictly restricted |

Separate Datastreams Per Environment

Always maintain separate datastreams for each environment:

```

Production Environment:

Datastream: "RetailStore - Production"

Used by: Live website, production app

Data goes to: Production sandbox, prod report suite

Staging Environment:

Datastream: "RetailStore - Staging"

Used by: Staging website, QA app

Data goes to: Staging sandbox, staging report suite

Development Environment:

Datastream: "RetailStore - Development"

Used by: Dev website, dev app builds

Data goes to: Dev sandbox, dev report suite

```
