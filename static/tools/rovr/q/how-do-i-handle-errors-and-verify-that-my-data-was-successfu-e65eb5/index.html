Handling Errors and Verifying Data Ingestion in Adobe Analytics

BDIA: Validation Before Upload (Strongly Recommended)

Adobe highly encourages using the Validate endpoint before establishing any BDIA workflow.

> If you do not first validate a file, you could end up with a combination of invalid and valid hits. The valid hits are processed while the invalid hits are silently discarded — causing major data quality issues including corrupted visit/visitor counts and broken attribution.

Why This Matters

  • Invalid rows are discarded without warning during live processing
  • Valid rows in the same file are still processed, so you won't get a clean failure — you'll get partial, corrupted data
  • Always validate first to catch issues before they affect your report suite

BDIA: Common Upload Errors and Fixes

File-Level Errors

These cause the entire upload to fail — no rows are processed. Fix the file and re-upload.

1. File Not in GZIP Format
  • Cause: The file was not properly compressed
  • Error response:

```json

{

"file_id": "3ae262c5-cdea-4bdb-a41c-fbd2c2004c4d",

"status_code": "REJECTED",

"processing_log": "An error occurred: Not in GZIP format",

"error": "No hits were found in the file."

}

```

  • Fix: Review your file creation process to ensure it properly compresses files using GZIP before upload
2. Missing Required Header Columns
  • Cause: The CSV file does not contain a required column such as a visitor ID header column
  • Fix: Ensure your CSV includes all required columns:
  • Visitor ID (e.g., `marketingCloudVisitorID`, `visitorID`, or `IPAddress`)
  • `pageName` (or a link equivalent)
  • `reportSuiteID`
  • `timestamp`
  • `userAgent`

Row-Level Errors

These affect individual rows only. Valid rows in the same file are still processed, which is why pre-validation is critical.

  • Individual invalid rows are silently discarded
  • The rest of the file continues processing
  • Without prior validation, you may not know which rows were dropped

Throttling and Volume Considerations

If you upload enough files in a short period to exceed your provisioned throttle limit:

  • Adobe does not reject the files outright
  • Instead, Adobe ingests uploaded files more slowly
  • This protects system stability but delays data availability

Best practices to avoid throttling:

  • Communicate your expected daily ingestion volume to Adobe in advance so appropriate hardware can be provisioned
  • Do not upload more than 1 file per second (use DIA if you require higher frequency)
  • Follow established Visitor Group guidelines to manage file send frequency limits

Expected Data Latency After Ingestion

Once files are successfully ingested, data does not appear in reports instantly:

| Data Age | Expected Reporting Latency |

|---|---|

| Timestamps < 24 hours old | Typically 20–50 minutes; some data available within minutes |

| Timestamps > 24 hours old | 2+ hours, depending on how old the data is |

Use these latency windows to determine when to check reports after a successful upload.

Verifying Successful Ingestion

1. Check the API Response

  • A successful upload returns a `file_id` and a `status_code` indicating acceptance
  • A rejected file returns `"status_code": "REJECTED"` with a `processing_log` explaining the failure

2. Use the Adobe Analytics Debugger (for AppMeasurement/Tag-based implementations)

  • Open Adobe's debugger or another packet monitor to validate that image requests are firing correctly with the right variables
  • This applies to client-side implementations rather than pure API-based ingestion

3. Check Reports in Adobe Analytics

  • After the appropriate latency window has passed, verify data appears in your report suite
  • Check key dimensions and metrics (page views, visits, eVars) to confirm expected values are present

4. Use Data Feeds for Raw Data Validation

  • Adobe offers Data Feeds containing all raw data for each day
  • Your organization can download and inspect the raw hit-level data to confirm specific records were ingested correctly
  • This is particularly useful for reconciling discrepancies between expected and reported data
