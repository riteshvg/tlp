The Scenario

Business Problem: A retail brand noticed that customers who browsed their website anonymously and later logged in were being treated as two separate profiles in Adobe Experience Platform. This caused:

  • Duplicate customer records in the profile store
  • Incorrect audience memberships — authenticated profiles missing behavioral data from anonymous sessions
  • Journey eligibility failures — customers not entering re-engagement journeys despite qualifying behavior
  • Inflated profile counts — reporting showing more profiles than actual customers

Step 1: Initial Investigation

Check the Symptoms

The first step was to identify the scope of the problem:

1. Navigate to Adobe Experience Platform

2. Go to Customer → Profiles

3. Search for a known test customer using their email address

4. Note the profile attributes and experience events attached

Observation: The authenticated profile (identified by email) showed no browsing history — all page view events were attached to a separate anonymous profile identified only by ECID.

Check Identity Graph

1. Navigate to Identities → Identity Graph

2. Enter the Email namespace and the customer's email address

3. Click View

Observation: The identity graph showed only the email identity with no ECID linked to it — confirming the anonymous and authenticated profiles were completely disconnected.

Step 2: Root Cause Analysis

Investigate Event Data

Using Adobe Experience Platform Assurance to inspect Web SDK events:

1. Navigate to Data Collection → Assurance

2. Create a new session and connect to the website

3. Perform a test login flow

4. Inspect the events fired during login

Finding: The login event payload was:

```json

{

"eventType": "userAccount.login",

"identityMap": {

"Email": [

{

"id": "customer@example.com",

"authenticatedState": "authenticated",

"primary": true

}

]

}

}

```

Problem Identified: The login event was sending only the Email identity — the ECID was missing from the identity map. Without ECID in the login event, AEP had no way to link the authenticated email identity to the anonymous ECID session.

Verify in Dataset

Using Query Service to confirm the issue:

```sql

SELECT

timestamp,

eventType,

identityMap,

_id

FROM

web_events_dataset

WHERE

eventType = 'userAccount.login'

AND timestamp > NOW() - INTERVAL '7 days'

ORDER BY

timestamp DESC

LIMIT 20

```

Query Result Confirmed: Login events had Email in `identityMap` but no ECID entry — confirming the root cause.

Step 3: Identity Namespace Configuration Review

Check Identity Settings

1. Navigate to Identities → Identity Settings

2. Review configured namespaces and their priorities

Findings:

| Namespace | Type | Priority | Unique Per Graph |

|---|---|---|---|

| ECID | Device | Low | No |

| Email | Person | High | Yes |

| Phone | Person | Medium | Yes |

| CRM ID | Person | High | Yes |

Observation: The namespace configuration looked correct — Email was set as high priority and unique per graph. The issue was purely in the data collection layer, not the identity configuration.

Check Schema Configuration

1. Navigate to Data Management → Schemas

2. Open the Web Events XDM Schema

3. Review identity field configuration

Finding: The schema had ECID configured as an identity field, but the Web SDK implementation was not including ECID in the login event's identity map.

Step 4: Graph Simulation to Understand Impact

Before making any fixes, used Graph Simulation to understand the expected behavior:

1. Navigate to Identities → Graph Simulation

2. Click Load → Existing graph

3. Enter the Email namespace and customer email value

Simulated the correct scenario:

Added both identities to the simulation:

  • Email: `customer@example.com` (authenticated, primary)
  • ECID: `12345678901234567890` (ambiguous)

Simulation Result: When both identities are present in the same event, AEP correctly merges them into a single graph — confirming the fix would resolve the issue.

Step 5: Implementing the Fix

Fix 1: Update Web SDK Implementation

Updated the login event to include both Email and ECID in the identity map:

Before (broken):

```javascript

alloy("sendEvent", {

xdm: {

eventType: "userAccount.login",

identityMap: {

Email: [{

id: "customer@example.com",

authenticatedState: "authenticated",

primary: true

}]

}

}

});

```

After (fixed):

```javascript

alloy("sendEvent", {

xdm: {

eventType: "userAccount.login",

identityMap: {

Email: [{

id: "customer@example.com",

authenticatedState: "authenticated",

primary: true

}],

ECID: [{

id: "12345678901234567890",

authenticatedState: "ambiguous",

primary: false

}]

}

}

});

```

> Note: The Web SDK automatically appends ECID to events when properly configured. The fix involved ensuring the `getIdentity()` API was called before sending the login event to retrieve the current ECID value.

Fix 2: Retrieve ECID Before Sending Login Event

```javascript

// Step 1: Get current ECID

alloy("getIdentity").then(function(result) {

const ecid = result.identity.ECID;

// Step 2: Send login event with both identities

alloy("sendEvent", {

xdm: {

eventType: "userAccount.login",

identityMap: {

Email: [{

id: "customer@example.com",

authenticatedState: "authenticated",

primary: true

}],

ECID: [{

id: ecid,

authenticatedState: "ambiguous",

primary: false

}]

}

}

});

});

```

Step 6: Validation After Fix

Validate with Assurance

1. Reconnect Assurance session to the updated website

2. Perform a test login

3. Inspect the login event payload

Confirmed: Login event now shows both Email and ECID in the identity map.

Validate Identity Graph

1. Navigate to Identities → Identity Graph

2. Search by Email namespace and customer email

3. View the updated graph

Result: Identity graph now shows Email and ECID linked in the same graph — profiles successfully merged.

Validate Profile in AEP

1. Navigate to Customer → Profiles

2. Search by email address

3. Review the Experience Events tab

Result: Authenticated profile now shows complete browsing history from anonymous sessions — full customer journey visible in a single profile.

Validate with Query Service

```sql

-- Verify login events now contain both identities

SELECT

timestamp,

eventType,

identityMap

FROM

web_events_dataset

WHERE

eventType = 'userAccount.login'

AND timestamp > NOW() - INTERVAL '1 hour'

ORDER BY

timestamp DESC

LIMIT 5

```

Query confirmed: All new login events contain both Email and ECID in the identity map.

Step 7: Monitor Results

Identity Dashboard Monitoring

After deploying the fix:

1. Navigate to Identities → Dashboard

2. Monitor Graphs with two or more identities metric

3. Track the increase in multi-identity graphs as existing profiles merge

Observation: Over the following 24-48 hours:

  • Multi-identity graph count increased as expected
  • Single-identity ECID-only profiles decreased
  • Profile count stabilized as duplicates merged

Journey Impact

After profile merging:

  • Re-engagement journeys started triggering correctly for qualified customers
  • Audience memberships updated to reflect complete behavioral history
  • Personalization improved as full customer context became available

Summary: Root Cause & Resolution

| Aspect | Details |

|---|---|

| Problem | Anonymous and authenticated profiles not merging |

| Root Cause | Login event missing ECID in identity map |

| Discovery Tool | Assurance + Identity Graph Viewer + Query Service |

| Simulation Tool | Graph Simulation to validate fix impact |

| Fix Applied | Updated Web SDK login event to include both Email and ECID |

| Validation | Assurance + Identity Graph Viewer + Profile UI + Query Service |

| Business Impact | Correct audience membership, journey eligibility, and personalization |

Key Lessons Learned

✓ Always include ECID in authenticated events – Linking anonymous and authenticated sessions requires ECID in every identity event

✓ Use Assurance for event debugging – Real-time payload inspection is the fastest way to find data collection issues

✓ Simulate before fixing – Graph Simulation validates expected behavior before production changes

✓ Query Service confirms at scale – SQL queries validate fixes across thousands of events

✓ Monitor post-fix – Track Identity Dashboard metrics to confirm merging is happening correctly

✓ Test the full journey – Validate that profile merging translates to correct journey and audience behavior
