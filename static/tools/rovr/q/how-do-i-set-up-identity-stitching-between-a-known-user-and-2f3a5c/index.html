How do I set up identity stitching between a known user and anonymous web visitor in AJO?

Setting Up Identity Stitching Between a Known User (CRMID) and Anonymous Web Visitor (ECID) in AEP/AJO

Identity stitching allows Adobe Experience Platform to unify a user's anonymous browsing behavior (tracked via ECID) with their authenticated profile (identified by a CRM ID), enabling real-time personalization and offer decisioning in Adobe Journey Optimizer.

How Identity Stitching Works

When a visitor arrives on your site anonymously, the AEP Web SDK assigns them an Experience Cloud ID (ECID). When that same visitor logs in, you capture their CRM ID (CRMID) and send it to AEP alongside the ECID. AEP's Identity Service links these two identities in the Private Identity Graph, building a unified profile that spans both anonymous and authenticated behavior.

Prerequisites

Before you begin, ensure you have:

  • Access to Adobe Experience Platform, Adobe Journey Optimizer, and Adobe Data Collection (Tags)
  • A working AEP Web SDK (Alloy.js) implementation via Adobe Tags
  • A Datastream configured and connected to AEP
  • Basic familiarity with XDM schemas, identity namespaces, and AEP datasets
  • JavaScript knowledge to capture login events on your webpage

High-Level Tutorial Steps

This solution is built across the following phases:

1. Import sample data

2. Create schema, dataset, and identity configuration

3. Create AEP Tags (Data Elements and Rules)

4. Send CRMID from the web page on login

5. Trigger a Journey Optimizer journey on user login

6. Test and validate the solution

Step 1: Import Sample Data

1. Prepare sample customer records that include both CRMID and associated profile attributes (e.g., email, name, ZIP code).

2. Upload this data to an AEP dataset that will serve as your profile dataset (record data).

3. Verify ingestion in AEP > Datasets to confirm records are appearing correctly.

Step 2: Create Schema, Dataset, and Identity Configuration

Create a Profile Schema (Record Data)

4. In AEP, navigate to Data Management > Schemas > Create schema.

5. Choose XDM Individual Profile as the base class.

6. Add the field groups needed for your use case (e.g., Personal Details, Demographic Details).

7. Add a custom field for CRMID and mark it as an identity field with a custom identity namespace (e.g., `CRMID`).

8. Mark CRMID as the Primary Identity on this schema.

9. Enable the schema for Real-Time Customer Profile.

Create an Event Schema (Time Series Data)

10. Create a second schema using the XDM ExperienceEvent base class.

11. Add the ECID field via the `identityMap` field group — this is automatically populated by the Web SDK.

12. Add a field for CRMID (non-primary identity) so login events carry both identifiers.

13. Enable this schema for Real-Time Customer Profile.

Create Datasets

14. Create a Profile dataset linked to your Profile schema.

15. Create an Event dataset linked to your ExperienceEvent schema.

16. Enable both datasets for Profile ingestion.

Step 3: Create AEP Tags (Adobe Data Collection)

Set Up the Tag Property

17. Navigate to Adobe Data Collection > Tags > New Property.

18. Name the property and associate it with your website domain.

Configure Extensions

19. Install the Adobe Experience Platform Web SDK extension.

20. Connect it to your Datastream (which should point to AEP and AJO).

Create Data Elements

21. Create a Data Element for ECID — the Web SDK populates this automatically.

22. Create a Data Element for CRMID — this reads the CRM ID from your login form, a data layer, or a cookie (e.g., `digitalData.user.crmId`).

Create Rules

23. Create a rule triggered on page load to fire a Web SDK `sendEvent` call — this captures anonymous browsing with ECID.

24. Create a second rule triggered on user login to fire a Web SDK `sendEvent` call that includes both ECID and CRMID in the `identityMap`.

Step 4: Send CRMID from the Web Page on Login

This is the critical step that initiates identity stitching.

25. On your login page, capture the CRMID after successful authentication (from your login API response, a data layer push, or a cookie).

26. Structure the `sendEvent` call to include the identity map with both identifiers:

```javascript

alloy("sendEvent", {

xdm: {

identityMap: {

ECID: [{

id: "<ecid_value>",

primary: false

}],

CRMID: [{

id: "<crmid_value>",

primary: true,

authenticatedState: "authenticated"

}]

},

eventType: "LoginEvent"

}

});

```

27. Ensure the `eventType` is set to `LoginEvent` — this is used later to trigger the AJO Journey.

28. Publish your Tag property changes and deploy to your website.

Step 5: Trigger an AJO Journey on User Login

Create an AJO Event

29. In Journey Optimizer, navigate to Administration > Configurations > Events.

30. Click Create Event and configure it to listen for events where `eventType = LoginEvent`.

31. Save the event.

Create the Journey

32. Navigate to Journey Management > Journeys > Create Journey.

33. Add the LoginEvent as the entry event (journey trigger).

34. Add an Email action node — configure it to send a personalized email to the user's profile email address.

35. Use profile attributes (e.g., name, offers) in the email content for personalization.

36. Set journey end conditions and publish the journey.

Step 6: Test and Validate the Solution

Validate Identity Stitching in AEP

37. Use the AEP Debugger browser extension to inspect Web SDK network calls — confirm both ECID and CRMID are being sent.

38. In AEP, navigate to Customer > Identities and use the Identity Graph Viewer.

39. Search for the CRMID or ECID — you should see both identities linked in the same identity graph node, confirming stitching is working.

Validate the Journey Trigger

40. Log in as a test user on your website.

41. In AJO, monitor Journey reporting or Live view to confirm the LoginEvent was received and the journey was triggered.

42. Check the test user's inbox for the personalized email.

Key Concepts Summary

| Concept | Description |

|---|---|

| ECID | Anonymous identifier assigned by the Web SDK to every visitor |

| CRMID | Known identifier captured at authentication (login) |

| Identity Graph | AEP's map linking ECID and CRMID to a single unified profile |

| identityMap | XDM structure used to pass multiple identities in a single event |

| authenticatedState | Set to `authenticated` when the user is logged in, `ambiguous` when anonymous |

| Persistent ID | The always-present ID used across all events (typically ECID) |

| Person ID | The authenticated ID present only on some events (typically CRMID) |

| Private Identity Graph | AEP's internal, org-specific map of stitched identity relationships |

Important Stitching Limitations to Be Aware Of

  • Stitching applies to event datasets only — lookup and profile datasets are not stitched.
  • Identity matching is case-sensitive — `Bob` and `BOB` are treated as two different people.
  • Stitching does not transform the identity field value — it uses the value exactly as it exists in the dataset.
  • For best results, a CRMID should appear on at least one event per ECID (i.e., at least one authenticated session per device).
