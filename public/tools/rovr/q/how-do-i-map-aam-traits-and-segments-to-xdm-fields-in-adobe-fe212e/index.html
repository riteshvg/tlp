How do I map AAM traits and segments to XDM fields in Adobe Experience Platform?

Mapping AAM Traits and Segments to XDM Fields in Adobe Experience Platform

This guide provides a complete, detailed walkthrough of how AAM traits and segments are mapped to XDM schemas and fields within Adobe Experience Platform.

Overview

When AAM data is ingested into AEP via the Adobe Audience Manager Source Connector, the data must be structured according to the Experience Data Model (XDM) standard. XDM is the foundational schema language of AEP that ensures all customer data is represented in a unified, consistent format.

Understanding the AAM-to-XDM Data Structure

Before mapping, it is essential to understand how AAM data types correspond to XDM concepts:

| AAM Data Type | XDM Equivalent | XDM Location |

|---|---|---|

| AAM Segments | Audience/Segment Membership | `xdm:segmentMembership` |

| Rule-based Traits | Profile Attributes | `xdm:traits` |

| Onboarded Traits | Record Data Fields | Custom XDM field groups |

| AAM UUID | Identity Namespace | `identityMap` |

| ECID | Identity Namespace | `identityMap.ECID` |

| Declared IDs | Identity Namespace | Custom identity namespace |

| Signals/Key-Value Pairs | Context Data | Custom XDM fields |

Prerequisites

Before beginning the mapping process, ensure you have:

  • ✅ System Administrator or Data Engineer role in AEP
  • ✅ AAM and AEP provisioned under the same IMS Organization
  • ✅ At least one XDM Schema already created in AEP
  • ✅ Familiarity with XDM Schema Editor in AEP
  • ✅ AAM Source Connector already configured (see previous guide)
  • ✅ Understanding of XDM field types (String, Integer, Boolean, Array, Map)

Phase 1: Preparing Your XDM Schema for AAM Data

Step 1: Navigate to the Schema Editor

1. Log in to Adobe Experience Platform.

2. From the left navigation, click Data Management > Schemas.

3. The Schemas workspace opens showing all existing schemas.

4. Click Create schema in the top-right corner.

5. Select the appropriate schema class:

| Use Case | Recommended Class |

|---|---|

| AAM Segment Membership (Profile data) | XDM Individual Profile |

| AAM Behavioral/Event data | XDM ExperienceEvent |

| AAM Prospect data | XDM Individual Prospect Profile |

6. Click Next to proceed to the Schema Editor.

Step 2: Add Required Field Groups to Your Schema

Field groups are pre-built collections of XDM fields. For AAM data, you need specific field groups:

1. In the Schema Editor, click Add next to Field groups in the left panel.

2. Search for and add the following field groups:

For Segment Membership Data:
  • Segment Membership Details — contains the `segmentMembership` map field
  • IdentityMap — contains the `identityMap` field for ECID and AAM UUID
For Profile/Trait Data:
  • Profile Personal Details — name, email, demographics
  • Profile Work Details — B2B-related fields
  • Demographic Details — age, gender, household data
For Media/Behavioral Data:
  • MediaAnalytics Interaction Details — media session and interaction fields
  • Session details information — contains redirect indicators and session data

3. Select each field group and click Add field groups.

4. Verify all selected field groups appear in the Schema Editor canvas.

Step 3: Add Custom Fields for AAM Traits

AAM rule-based and onboarded traits often require custom XDM fields since they represent your organization's unique data attributes.

1. In the Schema Editor, click the schema name at the top of the field hierarchy.

2. In the right-hand panel, click Add field (the + icon).

3. Configure each custom field:

Example: Mapping an AAM Trait "Has Purchased" to XDM

| Setting | Value |

|---|---|

| Field Name | `hasPurchased` |

| Display Name | Has Purchased |

| Type | Boolean |

| Description | AAM trait indicating purchase behavior |

| Required | No |

4. Click Apply to save the field.

5. Repeat this process for each AAM trait you want to map.

Step 4: Define Field Types for Different AAM Trait Categories

Different AAM trait types require different XDM field configurations:

Rule-Based Traits (Behavioral)

```

Field Type: Boolean or Integer

Example: "Visited Product Page" → Boolean (true/false)

Example: "Visit Count" → Integer

XDM Path: _tenantId.traits.visitedProductPage

```

Onboarded Traits (CRM/Offline Data)

```

Field Type: String or Integer

Example: "Customer Tier" → String ("Gold", "Silver", "Bronze")

Example: "Lifetime Value" → Number (decimal)

XDM Path: _tenantId.traits.customerTier

```

Algorithmic Traits (Model-Based)

```

Field Type: Number (0.0 to 1.0 score)

Example: "Propensity to Purchase Score" → Number

XDM Path: _tenantId.traits.propensityScore

```

Phase 2: Mapping AAM Segments to XDM Segment Membership

Step 5: Understand the segmentMembership XDM Structure

AAM segments map to the `xdm:segmentMembership` field in XDM, which is a Map data type. The structure is:

```json

{

"xdm:segmentMembership": {

"AAM": {

"segment_id_12345": {

"xdm:status": "realized",

"xdm:lastQualificationTime": "2024-01-15T10:30:00Z",

"xdm:version": "1",

"xdm:payload": {

"xdm:payloadType": "string",

"xdm:payloadStringValue": "AAM Segment Name"

}

}

}

}

}

```

Key Fields Within segmentMembership:

| Field | Type | Description |

|---|---|---|

| `xdm:status` | String | `realized`, `existing`, or `exited` |

| `xdm:lastQualificationTime` | DateTime | When the profile last qualified |

| `xdm:version` | String | Segment version number |

| `xdm:validUntil` | DateTime | Expiry of segment membership |

| `xdm:payload` | Object | Additional segment metadata |

Step 6: Map AAM Segment Status Values to XDM

AAM uses specific status terminology that maps to XDM segment membership statuses:

| AAM Status | XDM Status | Meaning |

|---|---|---|

| Active/Realized | `realized` | Profile just qualified for segment |

| Existing | `existing` | Profile continues to be in segment |

| Dropped/Exited | `exited` | Profile no longer qualifies |

Phase 3: Mapping AAM Identity Data to XDM

Step 7: Map AAM Identifiers to XDM identityMap

AAM uses several identifiers that must be mapped to the `identityMap` field in XDM:

1. In the Schema Editor, ensure the IdentityMap field group is added.

2. The `identityMap` field supports the following AAM identity mappings:

```json

{

"identityMap": {

"ECID": [

{

"id": "37657579838757832",

"authenticatedState": "ambiguous",

"primary": true

}

],

"AAM_UUID": [

{

"id": "64980308716806280",

"authenticatedState": "ambiguous",

"primary": false

}

],

"EMAIL": [

{

"id": "hashed_email@domain.com",

"authenticatedState": "authenticated",

"primary": false

}

]

}

}

```

Identity Namespace Mapping Table:

| AAM Identity | AEP Identity Namespace | Authentication State |

|---|---|---|

| ECID (MID) | `ECID` | `ambiguous` |

| AAM UUID (demdex) | `AAM_UUID` | `ambiguous` |

| Declared ID (CRM) | Custom namespace | `authenticated` |

| Hashed Email | `Email` | `authenticated` |

| Device ID (IDFA/GAID) | `IDFA` / `GAID` | `ambiguous` |

Step 8: Set Identity Fields as Primary Identifiers

1. In the Schema Editor, locate the `identityMap` field.

2. Click on the ECID identity field.

3. In the right panel, toggle Identity to ON.

4. Check the Primary Identity checkbox.

5. Select ECID from the Identity Namespace dropdown.

6. Click Apply.

> Important: Only one field can be set as the Primary Identity per schema. ECID is the recommended primary identity for web-based AAM data.

Phase 4: Using the Data Mapping Interface in AEP Sources

Step 9: Access the Mapping Interface During Source Setup

When setting up the AAM Source Connector dataflow, AEP provides a graphical mapping interface:

1. During the AAM Source Connector setup flow, after selecting traits and segments, you reach the Mapping step.

2. The mapping interface displays two columns:

  • Left column: AAM Source Fields
  • Right column: XDM Target Fields

Step 10: Create Field Mappings in the UI

Mapping AAM Traits to XDM Fields:

1. Click Add new mapping in the mapping interface.

2. In the Source field column, click the field selector icon.

3. Browse or search for the AAM trait field (e.g., `properties.trait_12345`).

4. In the Target field column, click the field selector icon.

5. Browse the XDM schema hierarchy to find your target XDM field.

6. Select the field and click Select.

7. The mapping row is now configured.

8. Repeat for all traits you want to map.

Example Trait Mappings:

| AAM Source Field | XDM Target Field |

|---|---|

| `properties.trait.gender` | `xdm:person.xdm:gender` |

| `properties.trait.age_range` | `xdm:person.xdm:birthYear` |

| `properties.trait.income_bracket` | `_tenantId.traits.incomeBracket` |

| `properties.trait.has_purchased` | `_tenantId.traits.hasPurchased` |

| `properties.trait.product_category` | `_tenantId.traits.productCategory` |

Step 11: Use Calculated Fields for Data Transformation

Some AAM fields require transformation before mapping to XDM. AEP's mapping interface supports calculated fields using functions:

1. Click Add calculated field in the mapping interface.

2. The calculated field editor opens.

3. Use built-in functions to transform data:

Common Transformation Examples:

Convert AAM timestamp to XDM DateTime format:

```

toDate(properties.lastQualificationTime, "yyyy-MM-dd'T'HH:mm:ss'Z'")

```

Convert AAM string to lowercase for email matching:

```

toLowerCase(properties.emailAddress)

```

Concatenate AAM fields:

```

concat(properties.firstName, " ", properties.lastName)

```

Convert AAM integer trait to Boolean:

```

properties.hasPurchased == 1 ? true : false

```

4. After defining the transformation, click Save.

5. Map the calculated field to the appropriate XDM target field.

Step 12: Validate Your Mappings

Before finalizing the dataflow:

1. Review each mapping row in the interface.

2. Check for warning icons — these indicate:

  • Type mismatch between source and target fields
  • Missing required target fields
  • Invalid transformation functions

3. Resolve all warnings before proceeding.

4. Click Preview to see a sample of how the data will look after mapping.

5. Verify the mapped values are correct in the preview.

Phase 5: Managing and Updating XDM Schemas for AAM Data

Step 13: Edit Existing Schema Relationships

AEP now supports updating schema relationships directly in the Schema Editor UI:

1. Navigate to Data Management > Schemas.

2. Open your AAM target schema.

3. In the Schema Editor, locate the relationship fields.

4. Click on a relationship field to:

  • Update the reference schema
  • Rename the relationship
  • Remove the relationship

5. Save changes — these propagate automatically to all datasets using this schema.

Step 14: Add New AAM Traits to an Existing Schema

As your AAM implementation evolves, you may need to add new traits to your schema:

1. Navigate to the existing AAM XDM schema.

2. In the Schema Editor, locate your custom field group.

3. Click Add field within the field group.

4. Define the new field following Step 3 above.

5. Click Save.

> Important: XDM schemas are append-only — you can add new fields but cannot remove or rename existing fields once data has been ingested. Plan your schema carefully before ingestion.

Step 15: Use ML-Assisted Schema Creation for AAM Data

AEP now offers ML-assisted schema creation to automatically generate XDM schemas from sample AAM data files:

1. Navigate to Data Management > Schemas.

2. Click Create schema.

3. Select Upload sample file instead of manually building.

4. Upload a sample CSV or JSON file exported from AAM containing trait data.

5. AEP's ML algorithm analyzes the file and:

  • Recommends appropriate XDM field types
  • Suggests standard XDM fields where applicable
  • Creates custom fields for AAM-specific attributes

6. Review the ML-generated schema in the Flat View editor.

7. Add, remove, or modify fields as needed.

8. Click Save to finalize the schema.
