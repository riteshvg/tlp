Overview

While Web SDK is powerful and elegant, real-world implementations frequently encounter pitfalls that impact data quality, performance, and user experience. Understanding the most common mistakes helps you avoid them.

Pitfall 1: ECID Loss at Login/Authentication

The Problem

This is the #1 issue in Web SDK implementations:

```

Scenario: Customer Journey

──────────────────────────

1. Customer visits website (anonymous)

→ Web SDK generates ECID: 98765432109876543210

→ Tracked as anonymous visitor for 10 minutes

2. Customer clicks "Login" button

→ Login event fired with Email identity

→ BUT ECID NOT included in identity map

→ Edge Network sees two separate identities:

• ECID: 98765432109876543210 (previous session)

• Email: jane@email.com (authenticated session)

→ Identity graph FAILS to link them

→ Profile merge DOESN'T HAPPEN

3. Result:

❌ Anonymous browsing history lost

❌ Customer appears as TWO profiles

❌ Journey abandonment recovery fails

❌ Personalization broken

❌ Attribution broken

```

Why This Happens

```

Common Implementation Error:

────────────────────────────

Incorrect sendEvent on login:

alloy("sendEvent", {

xdm: {

eventType: "userAccount.login",

identityMap: {

Email: [{

id: "jane@email.com",

authenticatedState: "authenticated",

primary: true

}]

// ❌ MISSING: ECID not included!

}

}

});

```

Web SDK has no way to connect the new Email identity to the existing ECID from the cookie because the ECID is not explicitly mentioned in this event.

The Fix

```javascript

// CORRECT: Include BOTH Email AND ECID in login event

alloy("getIdentity").then(function(result) {

var ecid = result.identity.ECID;

alloy("sendEvent", {

xdm: {

eventType: "userAccount.login",

identityMap: {

Email: [{

id: "jane@email.com",

authenticatedState: "authenticated",

primary: true // Email is primary when authenticated

}],

ECID: [{

id: ecid, // ✅ INCLUDE the ECID

authenticatedState: "ambiguous",

primary: false

}]

},

person: {

name: {

firstName: "Jane",

lastName: "Smith"

}

}

}

});

});

```

Why This Works:

```

With ECID Included:

──────────────────

1. Login event arrives at Edge Network with:

Email: jane@email.com

ECID: 98765432109876543210

2. Identity Service links them:

ECID: 98765432109876543210

↕ linked

Email: jane@email.com

3. Single unified identity graph created

4. Profile merge happens:

✅ Anonymous browsing history linked

✅ Single customer profile created

✅ All data consolidated

```

Prevention

Create a reusable Web SDK login rule in Tags:

```

Rule Name: "AEP - User Login"

EVENT:

Type: Custom Event

Event: user-login

ACTIONS:

Action 1:

Extension: Web SDK

Type: Get Identity

Store result in data element: "Current ECID"

Action 2:

Extension: Web SDK

Type: Send Event

XDM Data:

{

eventType: "userAccount.login",

identityMap: {

Email: [{

id: "%User Email%",

authenticatedState: "authenticated",

primary: true

}],

ECID: [{

id: "%Current ECID%",

authenticatedState: "ambiguous",

primary: false

}]

}

}

```

Pitfall 2: Missing Identity Namespace Configuration

The Problem

```

Scenario: Multiple Identity Namespaces

────────────────────────────────────────

Your implementation uses:

• Email (customer email)

• ECID (device ID)

• CRM_ID (customer database ID)

• Phone (phone number)

• Username (login handle)

Issue: If these namespaces are NOT configured as identities

in your XDM schema, they are treated as REGULAR DATA,

not as identity links.

Result:

❌ Graph collapse (unrelated profiles merged)

❌ Identity resolution fails

❌ Segment qualification doesn't work

❌ Journey entry conditions fail

```

Why This Happens

```

Common Mistake: Not Marking Namespaces as Identities

──────────────────────────────────────────────────────

In XDM Schema, identity fields must be configured with

"Identity" metadata. If they're just regular fields,

Identity Service can't use them for graph linking.

Incorrect Configuration:

{

"identityMap": {

"Email": [{

"id": "customer@email.com"

// Field NOT marked as identity in schema

// Treated as data, not identity

}]

}

}

Correct Configuration:

1. Schema includes "Identity Map" field group

2. Email namespace registered in AEP

3. Email field marked with identity metadata

4. Identity graph can now link Email to ECID

```

The Fix

Step 1: Register Identity Namespaces

1. Navigate to Adobe Experience Platform → Identities

2. Click Browse in left sidebar

3. Click Create identity namespace

```

Namespace Name: Email

Code: Email

Type: Individual

Description: Customer email address

Namespace Name: CRM_ID

Code: CRM_ID

Type: Individual

Description: Customer CRM database ID

```

Step 2: Add Identity Map Field Group to Schema

1. Navigate to Schemas in AEP

2. Open your Event Schema

3. Click Add → Field Groups

4. Search for "Identity Map"

5. Add the field group

Step 3: Verify Fields Marked as Identity

In schema editor:

```

identityMap (required)

└─ Email

└─ id (mark as Identity: checked ✅)

└─ CRM_ID

└─ id (mark as Identity: checked ✅)

└─ ECID

└─ id (mark as Identity: checked ✅)

```

Step 4: Send Events with All Namespaces

```javascript

alloy("sendEvent", {

xdm: {

eventType: "commerce.purchases",

identityMap: {

Email: [{

id: "jane@email.com",

authenticatedState: "authenticated",

primary: true

}],

CRM_ID: [{

id: "CRM-JANE-12345",

authenticatedState: "authenticated",

primary: false

}],

ECID: [{

id: "98765432109876543210",

authenticatedState: "ambiguous",

primary: false

}]

}

}

});

```
