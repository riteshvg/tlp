How do I set up a journey in AJO that uses both an event-based trigger and a scheduled trigger?

Setting Up a Journey in AJO with Both Event-Based and Scheduled Triggers

Based on the available documentation context, I can explain what is directly supported, and clarify an important architectural point about how AJO journeys work.

Important: AJO Journey Entry Types Are Mutually Exclusive

In Adobe Journey Optimizer, a single journey uses one entry type — either:

  • Event-triggered (Unitary Event or Business Event), or
  • Audience-based / Scheduled (Read Audience, which runs on a schedule or one-time)

A single journey cannot have both an event trigger and a scheduled trigger simultaneously on the same entry point. These are fundamentally different journey types configured at creation.

Option 1: Event-Triggered Journey

This triggers a journey in real time when a specific event occurs (e.g., a form submission, a login).

Prerequisites:

  • Access to Adobe Experience Platform
  • Access to Adobe Journey Optimizer
  • An XDM event schema and dataset configured in AEP
  • An email channel configuration (or other channel)

Steps:

1. Log in to Journey Optimizer

2. Navigate to Administration → Configurations

3. Click Manage on the Events card, then click Create Event

4. Define your custom event (e.g., a form submission event or login event), specifying the event type, schema, and key identifier fields

5. Save the event

6. Navigate to Journeys → Create Journey

7. Drag your custom event onto the canvas as the entry node

8. Add subsequent steps (email, wait, condition, etc.)

9. Publish the journey

Option 2: Scheduled (Read Audience) Journey

This triggers a journey for an entire audience on a defined schedule (daily, weekly, one-time).

Steps:

1. Log in to Journey Optimizer

2. Navigate to Journeys → Create Journey

3. In the journey properties, select Read Audience as the entry type

4. Choose your target audience

5. Set the schedule (one-time, daily, weekly, or monthly)

6. Add your journey steps (email, wait, condition, etc.)

7. Publish the journey

Option 3: Achieving "Both" via Complementary Journeys

If your use case requires both behaviors, the recommended approach is to run two separate journeys:

| Journey | Type | Purpose |

|---|---|---|

| Journey A | Event-triggered | Responds immediately when a user takes an action (e.g., form submission, login) |

| Journey B | Scheduled / Read Audience | Sends batch communications to a segment on a defined cadence |

You can coordinate these by:

  • Using audience membership updated by Journey A's event to feed into Journey B's scheduled read
  • Using suppression logic (conditions checking profile attributes) to prevent double-messaging users already touched by Journey A

| Capability | Event Journey | Scheduled Journey |

|---|---|---|

| Real-time trigger | ✅ | ❌ |

| Scheduled cadence | ❌ | ✅ |

| Single profile entry | ✅ | ✅ |

| Batch audience entry | ❌ | ✅ |
