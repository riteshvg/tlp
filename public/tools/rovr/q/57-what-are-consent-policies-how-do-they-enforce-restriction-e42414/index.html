57. What are Consent Policies? How do they enforce restrictions?

Consent Policies in Adobe Experience Platform

What are Consent Policies?

Consent Policies are rules that control profile-level activation to destinations based on individual customer consent and preference choices. They ensure that marketing communications respect what each customer has opted into or opted out of.

> In simple terms: A consent policy says — "Only activate this profile to this destination IF the customer has given consent for this specific purpose."

Consent policies are part of AEP's Data Governance framework and work alongside Data Usage Labels and Marketing Action Policies to create multi-layer compliance enforcement.

Important: Consent Policy Availability

⚠️ Consent policies and automatic enforcement are ONLY available for organizations that have purchased:

  • Adobe Healthcare Shield, OR
  • Adobe Privacy & Security Shield

Organizations without these add-ons can still use data governance policies (labels + marketing actions), but NOT consent policies.

Consent Policies vs. Data Governance Policies

These are two distinct policy types that both enforce governance:

| Aspect | Data Governance Policy | Consent Policy |

|---|---|---|

| What it checks | Data labels + marketing action | Customer consent attributes |

| Question asked | "Is this data allowed for this use?" | "Did the customer consent to this use?" |

| Example | "Cannot use PII (I1) for behavioral ads (C11)" | "Only email if customer consented to email marketing" |

| Enforcement | Blocks activation if violation found | Filters profiles based on consent |

| Required add-on | Included in all AEP/RTCDP | Healthcare/Privacy & Security Shield only |

Both Must Pass:

```

Attempt to Activate Audience to Destination

↓

╔════════════════════════════════════════╗

║ CHECK 1: DATA GOVERNANCE POLICIES ║

║ (Do labels allow this marketing use?) ║

╚════════════════════════════════════════╝

↓

✅ Pass → Continue

❌ Fail → BLOCK ACTIVATION

↓

╔════════════════════════════════════════╗

║ CHECK 2: CONSENT POLICIES ║

║ (Did customers consent?) ║

╚════════════════════════════════════════╝

↓

✅ Pass → ACTIVATION PROCEEDS

❌ Fail → EXCLUDE non-consenting profiles

```

How Consent Policies Work

Every consent policy has two logical components:

Part 1: The "IF" Condition

This defines WHEN the policy is evaluated — what triggers the policy check.

```

IF [condition is met]

```

The "IF" condition can be based on:

  • A specific marketing action being performed
  • Specific data usage labels present on the data
  • A combination of both using AND/OR logic

Example Conditions:

  • "IF marketing action = `Email Marketing`"
  • "IF marketing action = `SMS Marketing`"
  • "IF data has label `I1` AND action = `Behavioral Advertising`"
  • "IF action = `Direct Mail` OR action = `Push Notifications`"

Part 2: The "THEN" Consequence

This defines WHAT CONSENT ATTRIBUTES MUST BE PRESENT for a profile to be included in the activation.

```

THEN [require these consent attributes to be true]

```

The "THEN" clause checks for XDM Consent Data Fields in the customer's profile. These are standardized fields in the XDM schema:

Common Consent Attributes:

  • `consents.marketing.email.val = true`
  • `consents.marketing.push.val = true`
  • `consents.marketing.sms.val = true`
  • `consents.marketing.phone_calls.val = true`
  • `consents.marketing.direct_mail.val = true`
  • `personalization.content.val = true`
  • `personalization.marketing.preferred_channel.val`

Real-World Examples

Example 1: Email Marketing Consent Policy

Scenario: Marketer wants to activate an audience to an email marketing platform (Adobe Campaign).

Policy Configuration:

```

Policy Name: "Email Marketing - Consent Required"

IF: Marketing action = "Email Marketing"

THEN: Profile must have consents.marketing.email.val = true

```

How It Works:

1. Audience of 10,000 profiles is selected for activation

2. System checks each profile's consent status:

```

Profile A: email consent = true → ✅ INCLUDE in email campaign

Profile B: email consent = false → ❌ EXCLUDE from campaign

Profile C: email consent = unknown/missing → ❌ EXCLUDE (conservative)

Profile D: email consent = true → ✅ INCLUDE

... (10,000 profiles evaluated)

```

3. Only 7,500 consenting profiles are activated to Campaign

4. 2,500 non-consenting profiles are filtered out

Result:

```

Original Audience: 10,000 profiles

Profiles with email consent: 7,500

Profiles without email consent: 2,500 ← EXCLUDED

Final Activation: 7,500 profiles sent to email platform

```

What the Customer Sees:

  • Profile A receives emails (consented) ✅
  • Profile B does NOT receive emails (opted out) ✅
  • Profile C does NOT receive emails (never consented) ✅

Example 2: Multi-Channel Consent Policy

Scenario: Marketing team wants to activate a broad audience to a multi-channel orchestration platform (Adobe Journey Optimizer).

Policy Configuration:

```

Policy Name: "Multi-Channel Marketing - OR Logic"

IF: Marketing action = "Direct Marketing"

THEN: Profile must have AT LEAST ONE of:

  • consents.marketing.email.val = true

OR

  • consents.marketing.sms.val = true

OR

  • consents.marketing.push.val = true

```

How It Works:

Profiles are included if they consented to ANY of the channels:

```

Profile A:

  • email consent: true
  • sms consent: false
  • push consent: false

→ Result: ✅ INCLUDE (has email consent)

Profile B:

  • email consent: false
  • sms consent: true
  • push consent: false

→ Result: ✅ INCLUDE (has SMS consent)

Profile C:

  • email consent: false
  • sms consent: false
  • push consent: true

→ Result: ✅ INCLUDE (has push consent)

Profile D:

  • email consent: false
  • sms consent: false
  • push consent: false

→ Result: ❌ EXCLUDE (no channel consent)

Profile E:

  • email consent: unknown
  • sms consent: false
  • push consent: false

→ Result: ❌ EXCLUDE (conservative approach)

```

Real-World Application:

  • Journey Optimizer receives only profiles with at least one channel consent
  • JO respects individual channel preferences (email-only, SMS-only, etc.)
  • Complies with customer preferences while maximizing reach

Example 3: SMS Marketing Consent Policy

Scenario: Team wants to activate audience for SMS campaigns.

Policy Configuration:

```

Policy Name: "SMS Marketing - Strict Consent"

IF: Marketing action = "SMS Marketing"

THEN: Profile must have consents.marketing.sms.val = true

```

Audience Breakdown: 50,000 profiles

  • 35,000 with SMS consent = true
  • 10,000 with SMS consent = false
  • 5,000 with SMS consent = unknown/missing

Enforcement:

```

SMS Campaign Activation

├── Original audience: 50,000 profiles

├── Consent Policy applied: "SMS Marketing"

├── Profiles with SMS consent (true): 35,000 → ✅ ACTIVATED

├── Profiles without SMS consent (false): 10,000 → ❌ FILTERED

├── Profiles with unknown consent: 5,000 → ❌ FILTERED
