Overview

Debugging a Web SDK / Tags implementation in production requires a layered approach — using multiple tools to inspect different aspects of the data collection pipeline. Production debugging must be done carefully and non-invasively to avoid impacting live customer data.

```

┌─────────────────────────────────────────────────────────────────┐

│ DEBUGGING STACK │

│ │

│ Layer 1: Browser Console & Network Tab │

│ → Raw network requests, JavaScript errors │

│ │

│ Layer 2: Adobe Experience Platform Debugger │

│ → Visual inspection of Web SDK events │

│ │

│ Layer 3: Adobe Experience Platform Assurance │

│ → Real-time event validation │

│ │

│ Layer 4: Web SDK Debug Mode (alloy setDebug) │

│ → Verbose SDK logging in console │

│ │

│ Layer 5: AEP Dataset & Profile Validation │

│ → Verify data reached AEP correctly │

│ │

│ Layer 6: Tags Monitoring Hooks │

│ → Monitor rule execution events │

└─────────────────────────────────────────────────────────────────┘

```

Tool 1: Adobe Experience Platform Debugger

What It Is

The Adobe Experience Platform Debugger is a browser extension that provides a visual interface for inspecting all Adobe technology running on a page — including Web SDK events, Tags rules, Analytics beacons, and Target requests.

Installation

1. Install for Chrome: Search "Adobe Experience Platform Debugger" in Chrome Web Store

2. Install for Firefox: Search in Firefox Add-ons marketplace

3. Click the AEP Debugger icon in your browser toolbar to open

Switching Tag Environments

In production debugging, you can switch the Tags environment without touching the live site:

1. Open AEP Debugger

2. Navigate to Adobe Experience Platform Tags section

3. Click Connect a Tag Property

4. Select your property

5. Switch to Development or Staging environment

```

Environment Switcher:

─────────────────────

embed: //assets.adobedtm.com/launch-prod-abc123.min.js

Switch to: Development

Benefits:

→ Test new rules on LIVE production content

→ No impact on real visitors (only your browser)

→ Validate changes before publishing to production
```

Tool 2: Adobe Experience Platform Assurance

What It Is

AEP Assurance provides real-time, detailed validation of SDK events. Unlike the browser-based Debugger, Assurance connects directly to the Adobe platform and shows exactly what data is being received and processed.

Setting Up an Assurance Session for Production

1. Navigate to Adobe Experience Platform → Assurance

2. Click Create Session

3. Enter session details:

```

Session Name: "Production Debug - [Date]"

Base URL: https://www.retailstore.com

Connection Type: Web Link (for web)

QR Code (for mobile)

```

4. Copy the generated Assurance URL

5. Open the URL in your browser

6. Navigate the production website

7. Watch events populate in real-time

Tool 3: Web SDK Debug Mode

What It Is

Web SDK Debug Mode enables verbose console logging that shows exactly what the SDK is doing internally — event processing, identity management, consent checks, and network communication.

Enabling Debug Mode in Production (Your Browser Only)

Method 1: Via Browser Console (Recommended for Production)

```javascript

// Open browser developer tools console

// Type this command:

alloy("setDebug", { enabled: true });

// Output in console:

// [ALLOY] Debugging is enabled.

// This persists across page loads (stored in localStorage)

// Does NOT affect other visitors

```

Method 2: Via localStorage

```javascript

// Set in browser console

localStorage.setItem("alloy-debug", true);

// Refresh page to activate

```

Method 3: Via URL Parameter

```

// Add to any page URL

https://www.retailstore.com?alloy_debug=true

// Works immediately without console access

// Useful for sharing debug links with team members

```

Method 4: Via configure (NOT recommended for production)

```javascript

// Only use in development environments

// This enables debug mode for ALL visitors

alloy("configure", {

datastreamId: "YOUR_DATASTREAM_ID",

orgId: "YOUR_ORG_ID@AdobeOrg",

debugEnabled: false // ✅ Always false in production

});

```

Common Debug Mode Errors & Meanings

```

Error 1: "The instance has not been fully configured"

Meaning: configure() not called before sendEvent()

Fix: Ensure configure() fires before any other command

Error 2: "Invalid datastreamId"

Meaning: Datastream ID is incorrect or doesn't exist

Fix: Verify datastreamId in Web SDK extension configuration

Error 3: "Consent is out — event discarded"

Meaning: User has opted out, event not sent

Fix: Expected behavior if user rejected consent

Error 4: "Network request failed"

Meaning: Cannot reach adobedc.net / custom edge domain

Fix: Check firewall rules, CNAME configuration

Error 5: "XDM validation failed: missing required field"

Meaning: XDM payload doesn't match schema

Fix: Review XDM schema and fix missing required fields

```

Tool 4: Browser Developer Tools

Network Tab Inspection

Inspect Web SDK network requests directly:

```

Opening Network Tab:

─────────────────────

1. Open Chrome DevTools (F12)

2. Click "Network" tab

3. Filter by: "interact" or "collect" or "adobedc"

4. Refresh page and interact with site

Network Requests to Look For:

POST /ee/v2/interact → sendEvent calls

POST /ee/v2/collect → sendBeacon calls (page unload)

GET /ee/v2/acquire → Initial edge cluster negotiation

```

Console Tab for JavaScript Errors

```javascript

// Check for JavaScript errors in console:

Common Errors to Look For:

──────────────────────────

❌ "alloy is not defined"

Cause: Web SDK library didn't load

Fix: Check Tags embed code is on page

❌ "Cannot read property 'SKU' of undefined"

Cause: Data element reading undefined value

Fix: Check data layer is populated before rule fires

❌ "Refused to connect to 'adobedc.net'"

Cause: Content Security Policy blocking Web SDK

Fix: Add 'adobedc.net' to CSP connect-src directive

❌ "alloy: Unexpected response format"

Cause: Edge Network returned unexpected data

Fix: Check datastream configuration

```
