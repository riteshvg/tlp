Overview

Event duplication in Adobe Experience Platform refers to the occurrence of duplicate events during data ingestion. Understanding what causes duplicates is essential for maintaining data quality and accurate reporting.

Primary Causes of Event Duplication

1. Network-Related Issues

Network connectivity problems between client-side SDKs and the Edge Network are a common source of event duplication:

  • Internet Service Provider (ISP) failures – Temporary network outages
  • Mobile signal loss – Network disconnections on mobile devices
  • General network failures – Poor connectivity between customer and Edge Network
  • Public internet instability – Since SDK-to-Edge Network communication occurs over the public internet, network reliability impacts event delivery

Why this causes duplication: When a network issue occurs, the SDK may not receive confirmation that an event was successfully delivered. As a result, the SDK retries sending the event, which can result in multiple copies of the same event reaching the platform.

2. Internal Experience Platform Auto-Scaling Events

Adobe Experience Platform infrastructure occasionally triggers auto-scaling events that can cause duplication:

  • Cloud infrastructure volatility – Temporary system scaling to handle traffic
  • Data rebalancing – System redistributes data across infrastructure during scaling
  • System maintenance – Internal platform operations that may cause temporary duplicates

Why this causes duplication: During auto-scaling events, data may be temporarily duplicated across different nodes before being consolidated.

The "At-Least-Once" Processing Guarantee

The Adobe Experience Platform data collection layer is designed to support "at-least-once" processing semantics.

What This Means:

  • Guaranteed delivery – Every event is guaranteed to be delivered at least one time
  • Possible duplication – In rare, limited situations, events may be delivered more than once
  • No data loss – Events are never lost, but may be duplicated

> This is a fundamental design choice that prioritizes data completeness over perfect deduplication.

To learn more about message delivery guarantees and "at-least-once" semantics, see this article on message delivery guarantees.

When Event Duplication Occurs

Event duplication is rare and limited but can happen in these scenarios:

| Scenario | Cause | Frequency |

|---|---|---|

| Network retries | SDK retransmits after connection loss | Limited, occasional |

| Auto-scaling events | Infrastructure rebalancing | Rare |

| Edge Network timeouts | SDK doesn't receive confirmation | Limited |

| Mobile app crashes | App restarts and resends buffered events | Occasional |

| Duplicate SDK initialization | Multiple SDK instances running | Configuration issue |

Event Deduplication Options

To address potential duplicate events, Adobe Experience Platform and Journey Optimizer provide deduplication capabilities:

Deduplication at the Query Level

Adobe Experience Platform Query Service supports data deduplication using the `ROW_NUMBER()` function:

```sql

SELECT

*

FROM (

SELECT

*,

ROW_NUMBER() OVER (PARTITION BY eventId ORDER BY timestamp) as rn

FROM

your_event_dataset

)

WHERE

rn = 1

```

This query:

  • Uses `ROW_NUMBER()` to identify duplicate events by `eventId`
  • Keeps only the first instance (where `rn = 1`)
  • Removes all subsequent duplicates

Deduplication Scope

Deduplication can be applied at different scopes:

| Scope | Description |

|---|---|

| Global deduplication | Remove duplicates across all events |

| Per-user deduplication | Remove duplicates within a single user's event stream (using `identityMap`) |

| Per-transaction deduplication | Remove duplicates for specific transaction types |

Common Use Cases for Deduplication

1. Experience Events – Remove duplicate page views or interaction events

2. Purchase Events – Ensure purchase orders aren't counted twice

3. Metrics – Deduplicate metric calculations to avoid inflated counts

Deduplication for Experience Events Example

```sql

SELECT

timestamp,

eventType,

identityMap,

xdm.commerce.purchases.value,

ROW_NUMBER() OVER (PARTITION BY identityMap ORDER BY timestamp) as row_num

FROM

your_experience_events_dataset

WHERE

timestamp > NOW() - INTERVAL '30 days'

AND eventType = 'commerce.purchases'

QUALIFY

row_num = 1

```

Monitoring for Duplicates

Check for Duplicate Events Using Query Service

Query to identify duplicates:

```sql

SELECT

eventId,

COUNT(*) as duplicate_count

FROM

your_event_dataset

WHERE

timestamp > NOW() - INTERVAL '24 hours'

GROUP BY

eventId

HAVING

COUNT(*) > 1

ORDER BY

duplicate_count DESC

```

Query to see duplicate details:

```sql

SELECT

eventId,

timestamp,

identityMap,

eventType,

ROW_NUMBER() OVER (PARTITION BY eventId ORDER BY timestamp) as instance_number

FROM

your_event_dataset

WHERE

timestamp > NOW() - INTERVAL '24 hours'

ORDER BY

eventId,

timestamp

```

Monitor in Journey Optimizer

  • Check journey execution logs for duplicate event entries
  • Review journey step events dataset in AEP for repeat instances
  • Monitor message feedback events for duplicate message sends

Best Practices to Minimize Duplication

✓ Use unique event IDs – Assign a unique `eventId` to every event for easy deduplication

✓ Implement retry logic carefully – Configure SDK retry policies appropriately

✓ Monitor network conditions – Track SDK-to-Edge Network connectivity

✓ Test in poor network conditions – Simulate mobile signal loss and network failures

✓ Implement deduplication in queries – Use `ROW_NUMBER()` for analytics queries

✓ Monitor duplicate rates – Regularly check Query Service for duplicate patterns

✓ Use idempotent operations – Design downstream systems to handle duplicate events gracefully

Important Caution

> Deduplication at a person scope is evaluated by complete months in UTC time. A partial-month reporting window may not display all first or last instances if some occurred within the full month but outside of the reporting dates.

This is important when creating deduplication rules based on person-level data — ensure your time windows account for UTC month boundaries.

Summary

Event duplication in Adobe Experience Platform is:

  • Rare and limited – Occurs only in specific circumstances
  • Caused by network issues or infrastructure events – Not a design flaw
  • Guaranteed not to lose data – "At-least-once" processing ensures completeness
  • Manageable with deduplication tools – Query Service and deduplication logic can remove duplicates
  • Monitored and preventable – Best practices minimize duplication risk
