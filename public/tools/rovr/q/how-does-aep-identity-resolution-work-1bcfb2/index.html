AEP Identity Resolution

Adobe Experience Platform's identity resolution (also called Identity Stitching or Profile Merge) is the process of linking multiple identifiers across devices and channels to build a unified customer profile.

Core Concepts

identityMap & Primary Identity

  • Every XDM record can carry an identityMap — a map of namespace-to-identifier pairs
  • One identity must be designated as primary across all namespaces in a record
  • The primary identity is the key used for record matching and deletion operations
  • Records without a populated primary identity are skipped during processing (e.g., record delete work orders)

Identity Namespaces

  • Namespaces categorize identity types (e.g., Email, Phone, ECID, CRMID)
  • Namespaces must exist in Identity Service before they can be referenced
  • Each namespace can hold one or more identity descriptors per record

How Identity Resolution Works

1. Identity Graph Construction

  • As data is ingested, AEP builds an identity graph per individual
  • The graph links all known identifiers for a person across namespaces
  • For example, a CRMID, Email, and ECID can all be linked to the same person

2. Profile Merge

  • The Real-Time Customer Profile uses the identity graph to merge fragmented records into a single unified profile
  • Merge policies control which datasets and identity graphs are used when constructing a profile

3. Schema Requirements

  • The dataset schema must define a primary identity or identity map for identity resolution to function
  • If a primary identity field was added to a schema after data ingestion, previously ingested records are not eligible for identity-based operations
  • Secondary identities are not scanned for matching — only the primary identity is used

Data Partner Identity Resolution

AEP also supports third-party identity resolution through data partner integrations. For example:

Merkury Enterprise Identity Resolution

  • Provided by Merkle, this source connector enables recognition of digital visitors without cookies
  • Uses a person ID to combine everything an organization knows about an individual into a single comprehensive profile
  • Can incorporate:
  • Digital behaviors
  • Buying preferences
  • Name, email, physical address, device ID
  • Ingested data can be formatted as XDM JSON, XDM Parquet, or delimited files

Key Limitations to Be Aware Of

| Limitation | Detail |

|---|---|

| Primary identity required | Schemas must define a primary identity or identity map |

| Secondary identities not matched | Only primary identity is used for record matching |

| Pre-configuration data | Records ingested before identity was configured are not eligible |

| Unpopulated primaries skipped | Records missing primary identity metadata are ignored |

Identity Resolution in Practice (Record Deletes)

A practical example of how identity resolution is enforced is in record delete work orders:

  • Deletion targets the primary identity field exclusively
  • Up to 100,000 identities can be submitted per work order
  • Datasets without a schema-defined primary identity or identity map cannot be targeted

Summary

AEP identity resolution works by:

1. Collecting identifiers via identityMap in XDM schemas

2. Building an identity graph linking all known IDs for a person

3. Merging fragmented profile records into a unified Real-Time Customer Profile

4. Enforcing primary identity rules to ensure accurate matching and data governance
